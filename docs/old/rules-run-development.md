# Development Run Rules

この文書を、このリポジトリで開発シーケンスを実行する時に参照する正本として扱わなければならない。ここへ記述してよい内容は、開発単位、基本手順、テストと検証、完了条件、開発中の判断基準だけに限定し、それ以外を混在させてはならない。

rules コレクション全体の役割分担を判断する場合に限って [rules.md](../rules.md) を参照しなければならず、文書体系と履歴運用を判断する場合に限って [rules-update-documents.md](./rules-update-documents.md) を参照しなければならない。それ以外の論点ではこれらを参照してはならない。

## 開発プロセス

### 単位

- `開発セッション` は、ユーザーが同じ基準ブランチと GitHub 操作モードで継続作業すると合意している期間に限定しなければならず、`llm-cache/session-context.json` に保持する開発セッション情報を正本として扱わなければならない。これ以外の場所を正本として扱ってはならない。
- `開発シーケンス` は、1 件のユーザー指示を起点に開始し、履歴ローテーション、既存の開発セッション情報の参照、Issue 化、実装、検証、文書同期、コミット、Pull Request 作成までを含めて完了させなければならない。途中工程を恒久的に省略して完了扱いにしてはならない。`[反映済み]`

### 基本手順

- 1 周の開発は、`履歴ローテーションと開発セッション情報の確認`、`ユーザー指示の理解`、`Issue の用意と詳細化`、`先行テストで期待固定`、`実装と健康度点検`、`検証`、`文書同期`、`コミット`、`Pull Request の作成` の順で進めなければならない。正当な理由なく順序を入れ替えてはならない。
- 着手時は、ユーザー指示を読み取り、関連コードと関連文書を確認し、変更対象と影響範囲を把握する。`[反映済み]`
- 開発セッションの開始時は、まず現在のブランチをこのセッションの基準ブランチとして使うかをユーザーへ確認する。`[反映済み]`
- ユーザーが現在のブランチを使うと答えた場合は、そのブランチを `セッション限定の main` として扱わなければならず、そのセッション中の Issue ブランチ作成元と Pull Request の既定 base にしなければならない。`[反映済み]`
- ユーザーが現在のブランチを使わない場合は、通常の `main` へ移動し、`git pull --ff-only origin main` で最新へ更新できることを確かめてから作業に入る。`[反映済み]`
- 開発セッションの開始時は、現在の main ブランチ名と GitHub 操作モードを開発セッション情報として出力しなければならず、その後の Issue / Pull Request 作成や mode 分岐の参照元にしなければならない。`[反映済み]`
- 同じ開発セッション内の後続シーケンスでは、現在のブランチを基準ブランチとして使うかの問い合わせを繰り返してはならず、既存の開発セッション情報を参照しなければならない。`[反映済み]`
- 開発シーケンスの開始時は、`llm-cache/session-context.json` または対応する skill / script の出力を確認し、現在の session main と mode を取り違えずに Issue ブランチ作成元と Pull Request の既定 base を決めなければならない。`[反映済み]`
- チャット欄からの指示で開発シーケンスを始める場合も、まずその指示を原文として Issue を作成しなければならず、以後は Issue 駆動の開発として扱わなければならない。Issue なしで着手してはならない。`[反映済み]`
- 開発シーケンスの開始時に日付跨ぎや過去日の履歴が `*-latest.md` に残っていた場合は、[rules-update-documents.md](./rules-update-documents.md) に従い、[scripts/rotate-history](../../scripts/rotate-history) を使って過去日分だけを `*-log.md` へ移し、当日分を `*-latest.md` に残したまま運用を継続する。`[反映済み]`
- 履歴ローテーションは開発シーケンスの開始直後に完了させ、当日分と過去日分が同じ `*-latest.md` に混在したまま次の更新を続けてはならない。`[反映済み]`
- LLM が作業中に一時ファイルを置く必要がある場合は、プロジェクト直下の `llm-temp/` を使い、`/tmp` など共有領域へ安易に退避しない。
- Git 管理下のファイルを移動する場合は、まず `git mv` だけで移動し、その移動単体をコミットしてから、リンク、参照、文書内の相対パス、コード側の参照を次の変更セットで修正する。

### Issue 駆動の進め方

- チャット欄から着手したタスクでは、作業開始前にユーザー指示を原文として残した Issue を作成する。`[反映済み]`
- Issue の Assignee / Project と Pull Request の Assignee の既定値は、rules へ直書きしてはならず `llm-cache/` と secrets から解決しなければならない。Pull Request を Project へ自動登録する前提で運用してはならない。
- GitHub 操作モードは secrets の `operationMode` で管理し、rules には特定プロダクトの固定モードを埋め込まない。
- `user` モードでは、Issue / Pull Request の repo 操作は GitHub App で行わなければならず、Projects 操作は `gh` で行わなければならない。
- `organization` モードでは、Issue / Pull Request の repo 操作も Projects 操作も GitHub App で行わなければならない。
- Issue の既定 Assignee と Project 一覧、および Pull Request の既定 Assignee は `llm-cache/` 配下の local cache を正本として参照しなければならず、必要項目が無い時は処理を中断し、コメントを出力してユーザーへ問い合わせなければならない。推測で補完してはならない。
- rules へプロダクト固有値を持ち込む指示、または既存ルールに反する指示を受けた場合は、その内容を Issue コメントへ記録して停止し、次の指示を待つ。
- チャット欄から作成した Issue は、元の指示を `原文` として残したうえで、Description にはチェックボックス付き ToDo のみを追記しなければならない。背景、目的、スコープ、実施タスク、完了条件などの詳細化本文は Issue コメントで追記し、そのコメント以後の整理結果を実装上の正本として扱わなければならない。
- チャット起点で作成した Issue は、詳細化が完了した時点で通常の Issue 起点タスクと同じプロセスへ合流させなければならず、以後は区別してはならない。
- Issue に基づいて着手するタスクでは、実装前に対象 Issue の Description を読み、作業単位、完了条件、非対象を確認してから着手する。
- `Issue-x を詳細化せよ` と指示された場合は、ユーザーから与えられた元の指示自体を残さなければならず、Description にはチェックボックス付き ToDo だけを追記しなければならない。背景、目的、スコープ、実施タスク、完了条件などの構造化内容は Issue コメントで追加し、そのコメントを以後の実装上の正本として扱わなければならない。
- Issue の詳細化中に blocker が見つかった場合は、その時点で Issue コメントへ理由、確認した内容、現在の状況を書き残して停止する。
- Issue を詳細化する時は、本文だけでなくタイトルも内容に見合う具体度へ更新し、一覧から見ても作業対象が判別できる状態を保つ。
- Issue に着手したら、実装開始前に対象 Issue に対応する作業ブランチを作成しなければならず、そのブランチ名を対象 Issue のコメントへ記録しなければならない。記録時は [scripts/register-issue-branch](../../scripts/register-issue-branch) を正規入口として使わなければならず、別経路で表記揺れしたコメントを残してはならない。進行状態の管理は Issue ラベルではなく GitHub Projects や Pull Request の関係で追跡しなければならない。
- 通常の `main` を使う場合でも `セッション限定の main` を使う場合でも、実装は基準ブランチへ直接積んではならず、必ず対象 Issue 用の作業ブランチで進めなければならない。
- Issue に実施タスクの ToDo がある場合は、ToDo を 1 つ完了するごとに対応する変更をコミットし、変更とタスクの対応関係が履歴から追える状態を保つ。
- ToDo が複数ある場合でも、複数ステップを 1 つのコミットへまとめず、各ステップごとに履歴を分ける。
- 実装中は、意味のある変更がまとまるたびにコミットし、途中経過が再現できる粒度で履歴を残す。
- タスクを実施できないと判明した場合は、新規実装や派生作業へ進まず、Issue のコメントに理由、確認した内容、現在の状況を書き残して処理を中断する。
- Issue コメントで blocker や確認事項の問答を行ったタスクを完遂した場合は、最後に「どの問題をどう処置して完了へ進めたか」を Issue コメントへ追記し、Issue 上から最終判断が辿れる状態を保つ。
- タスク完了時は、関連する検証と文書同期を終えたうえで、merge 先が通常の `main` か `セッション限定の main` かを問わず必ず Pull Request を作成しなければならない。Pull Request には Issue と同じ解決経路で Assignee を設定しなければならないが、Project を自動登録してはならない。LLM は Issue を直接 close してはならない。
- GitHub Project に `LLM所要時間` の Number フィールドがある場合は、開発シーケンスの終盤で実測した分数を対応する Issue 項目へ反映する。フィールドが無い場合は対応する skill / script で作成してから更新する。

### テストと検証

- 機能追加では、実装前に期待する振る舞いを表すテストを先に追加し、そのテストで失敗を確認してから実装へ進む。
- 不具合対応では、想定原因候補ごとに再現または非再現を担保するテストを先に追加し、そのテストで失敗を確認してから修正へ進む。
- 実装中は、責務境界、複雑度、リソースの作用域、状態管理の健全性を同時に点検し、悪化する変更を放置しない。
- 実装言語ごとの formatter / lint と複雑度しきい値を判断する場合は [rules-change-languages.md](./rules-change-languages.md) を正本として参照しなければならない。これに該当しない場合は参照してはならない。
- 検証では、変更内容に応じたテストと build 確認を行い、`error 0` かつ `warning 0` を確認する。
- 検証が最初の 1 回で通らなかった場合は、修正ループ中の再確認を関連テストと影響が及ぶはずのテストへ限定し、それらが通った後で最後に全体テストを実行する。
- 最終の全体テストでは、`scripts/collect-metrics.sh` を使って build 時間、全体 test 時間、起動性能、`docs/metrics/metrics-test.md` を同じ 1 回の全体実行から取得し、同じ全体テストを別スクリプトで重複実行してはならない。
- `tools`、`skills`、`scripts` だけを変更した場合は、アプリ本体の build や test は実施せず、対象ツールの構文確認と代表的な 1 経路の実行確認で検証する。

### 文書同期と完了条件

- 文書同期の対象と更新方法は [rules-update-documents.md](./rules-update-documents.md) に従う。
- 開発シーケンスの最後では、文書更新へ入る直前に [rules.md](../rules.md) を再度読み返し、長期スレッドやコンテキスト圧縮の後でもルール外の更新を混ぜていないことを確認する。
- ソースコード変更を含む場合は、ユーザーから明示的に止められていない限り、テスト、文書更新、計測記録を含めて最後にコミットまで実施する。
- ドキュメントのみを変更した場合も、ユーザーから明示的に止められていない限り、その開発シーケンスに必要な文書更新と当日ログ更新を行い、最後にコミットまで実施する。
- 細かいコミットが積み上がること自体は許容し、未確定の複数シーケンスを 1 つの大きなコミットへまとめることより、各開発シーケンスをコミットで確定することを優先する。
- コミット対象は、その時点で完了条件を満たした変更セット全体にしなければならず、コミットメッセージは日本語で記述しなければならない。
- Issue 駆動で進めたタスクは、Issue、ブランチ、コミット、Pull Request の対応関係が追跡できる状態で完了させる。

## 開発中の判断基準

- この節では、実装中の局所的な書き方だけでなく、文書整理、秘密情報の扱い、健全性観測、試行回数の自己診断、しきい値超過時の停止判断まで判断基準へ含めなければならない。局所実装の話だけに縮退させてはならない。

- テストのためだけの分岐は、本番実装へ広げずテスト支援レイヤーへ閉じ込める。
- 既存文書の章構成を尊重しなければならず、関連するルールは適切な節へ整理して記述しなければならない。
- 秘密情報がチャットへ貼られた場合は、その値を以後の記録文書へ転記してはならず、必要ならマスクして扱わなければならない。
- `llm-temp/` 配下の一時ファイルは、ユーザーが後から処理経路を見直すための痕跡として扱わなければならず、LLM の判断で自動削除してはならない。

### テスト原則

- テスト戦略は、まず domain / logic 側で固定できる契約を `unit test` または軽量統合テストへ寄せ、画面層でしか守れない契約だけを `UI test` へ残す方針で組み立てる。

#### 役割分担

- テストは `unit test` と `UI test` の責務を分ける。
- テストは後追いの確認手段としてだけ扱ってはならず、仕様を明確化し、仕様競合や成立不能条件を早期検出するための正本の一部として扱わなければならない。
- 純粋なルール、並び順、parser、resource 読込は `unit test` で担保する。
- 画面遷移、主要導線、操作体験、モック経路は `UI test` で担保する。
- UI テストへ載せるのは、`ユーザー導線として壊れて困る契約` と `画面層でしか確認できない契約` に限定する。

#### 寄せ方

- UI の見た目、画面遷移、画面内フィードバック、操作導線を変更していない場合は、影響箇所が UI 層にあっても UI テストを省略してよく、その代わりに unit test や軽量統合テストで変更契約を固定する。
- 宣言的 UI 基盤や標準フレームワークへ委ねている振る舞いは、独自ロジックが無い限り UI テストで過剰に縛らず、まず pure logic や軽量統合テストへ寄せる。
- UI テストの所要時間は品質コストの一部として扱い、削減時は `代わりに何を unit test で固定したか` を明確にしたうえで、必要な導線テストは残す。
- 非同期ロード後の件数、内部状態の集約結果、並び順の細部、サマリー表示の内容のように UI 以外でも検証できる内容は、まず `unit test` または軽量な統合テストへ寄せる。
- UI 導線から起動される機能でも、domain / logic 側へ切り出せる判定や状態遷移がある場合は、まずその層でロジックテストを追加し、その後に UI を写像として確認する。

### 健全性の観測

- 機能追加や障害修正では、動作確認だけでなく `責務境界`、`複雑度`、`リソースの作用域`、`状態管理の健全性` を同時に点検する。
- 点検の起点として既存の集約観測を使い、少なくとも `ファイル行数`、`関数行数`、`1ファイル内の型数`、`公開状態量` を継続観測する。
- 開発プロセスの健康度として、`LLM試行回数` と `同一論点の再試行回数` を毎回自己診断する。

### 試行回数の数え方

- `LLM試行回数` は、1 つのユーザー指示に対して `仮説 -> 実装またはテスト -> 失敗 -> 修正` を何巡したかの回数として数える。
- `同一論点の再試行回数` は、同じ失敗原因に対してテストや実装の修正をやり直した回数として数える。
- TDD のために先に追加した `失敗するべき新規テスト` を、その後の実装で通した `赤 -> 緑` の 1 往復は、`LLM試行回数` と `同一論点の再試行回数` のどちらにも含めない。
- 目安として、`LLM試行回数 >= 4`、または `同一論点の再試行回数 >= 3` に達した場合は、開発速度と精度のバランスが崩れている可能性が高いとみなす。
- 自己診断がしきい値に抵触した場合は、設計上の問題かプロセス上の問題かを切り分けるまで新規実装を中止し、その旨と観測内容をユーザーへ返して次の指示を待つ。

### 健全性のしきい値

- `Swift ファイル行数` は原則 `500` 行未満を維持し、`900` 行以上は分割なしに進めてはならない。
- `関数行数` は原則 `60` 行未満を維持し、`80` 行以上は分割または責務移譲なしに進めてはならない。
- `1ファイル内の型数` が `12` を超える場合は、モデル、永続化、外部通信、UI 補助などが混在していないか見直す。
- `ObservableObject` や coordinator の `@Published` 状態は `8` 個前後を目安とし、増え続ける場合は別 service、store、または画面ローカル state へ戻すことを検討する。

### 秘密情報と外部連携

- API キーやトークンなどの秘密情報は、リポジトリへコミットせず、ローカル専用かつ ignore 済みの設定ファイルから注入する。
- 外部 API の認証情報は、可能な限り URL クエリへ載せず、ログや履歴へ残りにくい経路を優先する。
- 外部 API 用のキーは用途ごとに分けなければならず、利用 API やアプリ種別で制限できる前提で扱わなければならない。
