# Development Principles

この文書は、HelloWorld に閉じない開発原則と知識ベースを記述する正本である。ここでは、言語やプロジェクトが変わっても再利用しやすい責務分離、テスト戦略、評価観点、文書設計の原則を扱う。

文書群全体での役割分担は [document-roles.md](./document-roles.md)、文書運用ルールは [document-operations.md](./document-operations.md) を参照する。HelloWorld 固有の意思決定ルールや運用は [rules.md](./rules.md) を参照する。

## 使い分け

- その内容を守らないと当該プロダクトが壊れるなら、`rules.md` へ置く。
- その内容を知っていると判断品質が上がるが、別プロダクトにも持ち運べるなら、`principles.md` へ置く。
- クラス名や型名が出る内容は、原則として `design.md` へ置く。
- ユーザー操作や画面導線に依存する内容は、原則として `spec.md` へ置く。
- レイヤ構造、依存方向、データフローの形、責務の原則を規定する内容は、原則として `architecture.md` へ置く。

## 責務分離

- 画面は表示責務に集中し、ルール判断や I/O を直接持たない。
- ルールは UI から切り離し、再利用可能な pure logic として扱う。
- UI に見える機能であっても、その連鎖の中に UI と無関係に成立すべき判定、状態遷移、キャッシュ更新、外部通信結果の統合が含まれる場合は、それらを domain / logic 側へ置き、UI はその状態の写像として組み立てる。
- 永続化や外部通信は、UI から直接触らず、中間層を介して扱う。
- 端末差分やレイアウト差分は、機能差分ではなく表現差分として扱う。
- View は一時的な UI 状態、アニメーション状態、選択状態、ダイアログ状態を持ってよく、MVVM を理由にそれらを機械的に ViewModel へ押し上げてはならない。
- ViewModel または coordinator は `1 画面` もしくは `1 機能の orchestration` に責務を寄せ、画面描画専用の細かな値変換や単純な表示状態まで過剰に抱え込まない。
- 依存方向は `View -> ViewModel/coordinator -> service/use case -> Store or Infrastructure` を原則とし、内側の層が外側の UI 都合へ依存してはならない。
- MVVM を守るためだけの protocol 分割、値の受け渡しラッパー、薄い中継層は増やさず、代わりに `pure logic`、`store`、`external service` へ意味のある単位で分割する。
- 固定パス、永続ファイル、検索キャッシュ、秘密情報解決のような `スコープの広いリソース` は、専用の `Paths` / `Store` / `Service` 型へ閉じ込め、View や汎用 model ファイルへ散らしてはならない。

## テスト原則

### 役割分担

- テストは `unit test` と `UI test` の責務を分ける。
- テストは後追いの確認手段ではなく、仕様を明確化し、仕様競合や成立不能条件を早期検出するための正本の一部として扱う。
- 純粋なルール、並び順、parser、resource 読込は `unit test` で担保する。
- 画面遷移、主要導線、操作体験、モック経路は `UI test` で担保する。
- UI テストへ載せるのは、`ユーザー導線として壊れて困る契約` と `画面層でしか確認できない契約` に限定する。

### 寄せ方

- UI の見た目、画面遷移、画面内フィードバック、操作導線を変更していない場合は、影響箇所が UI 層にあっても UI テストを省略してよく、その代わりに unit test や軽量統合テストで変更契約を固定する。
- データバインディングや Adaptive UI のように OS と宣言的 UI 基盤へ委ねている振る舞いは、アプリ独自ロジックが無い限り UI テストで過剰に縛らず、まず pure logic や軽量統合テストへ寄せる。
- UI テストの所要時間は品質コストの一部として扱い、削減時は `代わりに何を unit test で固定したか` を明確にしたうえで、必要な導線テストは残す。
- 非同期ロード後の件数、内部状態の集約結果、並び順の細部、サマリー表示の内容のように UI 以外でも検証できる内容は、まず `unit test` または軽量な統合テストへ寄せる。
- UI 導線から起動される機能でも、domain / logic 側へ切り出せる判定や状態遷移がある場合は、まずその層でロジックテストを追加し、その後に UI を写像として確認する。

### 観測と検証

- ビルド確認では `error` だけでなく `warning` も 0 件であることを確認し、警告を残したまま完了扱いにしてはならない。
- 機能追加、不具合修正、機能削除のいずれでも、先に追加したテストや既存テストを変更後の仕様へ合わせて更新し、古い仕様を表すテストを放置してはならない。
- ソースコードを変更した場合は、変更内容に応じたテストを実行し、通過を確認してからコミットする。
- 起動性能や検証コストの観測は、可能な限りコマンドで再取得できる形を優先する。

## 評価観点

### 健全性の観測

- 機能追加や障害修正では、動作確認だけでなく `責務境界`、`複雑度`、`リソースの作用域`、`MVVM の健全性` を同時に点検する。
- 点検の起点として `health_barometer` のような集約観測を使い、少なくとも `ファイル行数`、`関数行数`、`1ファイル内の型数`、`公開状態量` を継続観測する。
- 開発プロセスの健康度として、`LLM試行回数` と `同一論点の再試行回数` を毎回自己診断する。

### 試行回数の数え方

- `LLM試行回数` は、1 つのユーザー指示に対して `仮説 -> 実装またはテスト -> 失敗 -> 修正` を何巡したかの回数として数える。
- `同一論点の再試行回数` は、同じ失敗原因に対してテストや実装の修正をやり直した回数として数える。
- TDD のために先に追加した `失敗するべき新規テスト` を、その後の実装で通した `赤 -> 緑` の 1 往復は、`LLM試行回数` と `同一論点の再試行回数` のどちらにも含めない。
- 目安として、`LLM試行回数 >= 4`、または `同一論点の再試行回数 >= 3` に達した場合は、開発速度と精度のバランスが崩れている可能性が高いとみなす。
- 自己診断がしきい値に抵触した場合は、設計上の問題かプロセス上の問題かを切り分けるまで新規実装を中止し、その旨と観測内容をユーザーへ返して次の指示を待つ。

### 健全性のしきい値

- `Swift ファイル行数` は原則 `500` 行未満を維持し、`900` 行以上は分割なしに進めてはならない。
- `関数行数` は原則 `60` 行未満を維持し、`80` 行以上は分割または責務移譲なしに進めてはならない。
- `1ファイル内の型数` が `12` を超える場合は、モデル、永続化、外部通信、UI 補助などが混在していないか見直す。
- `ObservableObject` や coordinator の `@Published` 状態は `8` 個前後を目安とし、増え続ける場合は別 service、store、または画面ローカル state へ戻すことを検討する。

## 文書設計の原則

- 文書は役割ごとに責務を分け、同じ意味の節名は文書間でそろえる。
- 文書の内容が詳細化しすぎた場合は、上位方針を残して適切な下位文書へ移す。
- 追記メモを末尾へ積み上げるのではなく、既存の章構成へ統合して重複と矛盾を避ける。
- 実装と文書にずれが出た場合は、どちらが正しいか確認してから揃える。
