# オートリフレッシュログ調査レポート 2026-04-15

この文書は、`llm-temp/app.log` に追加されたログを使って、Mac 上のオートリフレッシュ不具合の原因調査をどのように収束させたか、今回の例で何が分かったか、検知した別件は何か、そして手動更新がどう振る舞っていたかを再利用可能な形で整理した参照資料である。正本の方針や責務は [AGENTS.md](../../AGENTS.md) と、必要に応じて `skills/` 配下の手順文書を参照する。

## 暫定結論

- 収束に最も効いたのは、`起動したか`、`sleep に入ったか`、`sleep から戻ったか`、`dispatch したか` をログ境界で追い、`停止` 仮説を順番に潰したことである。
- 今回の更新済みログでは、オートリフレッシュループは起動しており、`sleep -> wake -> snapshot -> dispatch -> cycle finish` の反復も確認できたため、「最初から動いていない」や「途中で完全停止した」とは言えない。
- 主な課題として残ったのは、初回の手動更新が長いこと、そしてログ末尾が最後の `sleeping` の直後で切れているため、最終スリープ後の起床観測がまだないことである。
- 次回は、`manual refresh` の内部段階をさらに分けて計測し、`sleep` 後の起床ログを終端まで押さえるのが最優先である。
- 別件として、サムネイルローカルファイル欠損による `NSURLErrorDomain -1100` が大量に出ており、これはオートリフレッシュの生死とは別に切り分けるべきノイズまたは障害候補である。

## 目的

- 目的1: ここまでのログを使って、原因調査をどのように収束させるべきかを整理する。
- 目的2: 今回の例で、どのログが効き、どの仮説が外れたかを明確にする。
- 目的3: 検知した別件を、オートリフレッシュ本体と混同しない形で記録する。
- 目的4: 手動更新の実際の挙動を、今後の再調査で使える粒度で残す。

## 実施の流れ

### 1. まず「起動していない」仮説を潰した

- `home_auto_refresh_task_started`
- `home_auto_refresh_manual_refresh_started`
- `refresh_cache_manual_started`
- `auto_refresh_loop_start_requested`
- `auto_refresh_loop_entered`

  これらの境界ログが並んでいるかを確認し、ループそのものが立ち上がっているかを見た。
- この段階で、少なくとも「起動処理が呼ばれていない」線は弱くなった。

### 2. 次に `sleep` と `wake` を追った

- `auto_refresh_loop_snapshot_evaluated due_channels="0"`
- `auto_refresh_loop_sleeping delay_ms="487899"`
- その後の `auto_refresh_loop_snapshot_evaluated due_channels="81"`
- さらに `auto_refresh_loop_sleeping delay_ms="550983"`
- その後の `auto_refresh_loop_snapshot_evaluated due_channels="92"`
- 最後に `auto_refresh_loop_sleeping delay_ms="551029"`

  という並びを確認し、`sleep` へ入ったあとに実際に戻ってきているかを見た。
- ここで、ループが死んでいるのではなく、待機と再評価を繰り返していると分かった。

### 3. さらに `dispatch` と `cycle finish` を見た

- `auto_refresh_loop_dispatching`
- `auto_refresh_cycle_started`
- `refresh_cycle_started`
- `refresh_cycle_finished`
- `auto_refresh_cycle_finished`

  の組が複数回出ているため、単なる再評価だけでなく、実際に更新処理へ進んでいることが確認できた。
- `due_channels` が 0 の回と 81 / 45 / 92 / 34 の回が混在しているため、状態遷移は固定停止ではなく、都度の判定で変わっている。

### 4. 手動更新の長さを見た

- 初回の大きな手動更新では、`298` チャンネルを対象にした更新が長く続いた。
- その後の自動循環でも、`81` 件で約 33 秒、`45` 件で約 17 秒、`92` 件で約 37 秒、`34` 件で約 13 秒ほどの更新区間が見えている。
- つまり、体感で「止まったように見える」原因として、更新自体の長さは十分ありうる。

## 評価

### 調査収束の方法として有効だった点

- 境界ログを先に決めたことが効いた。
- `started`、`entered`、`snapshot`、`sleeping`、`dispatching`、`finished` の順で追うと、停止なのか待機なのかを切り分けやすい。
- 反対に、ログの本文や周辺ノイズから先に読むと、停止仮説に引きずられやすい。

### 今回の例の詳細

- ログは、`home_auto_refresh_task_started` から始まり、初回の手動更新完了後に `home_auto_refresh_background_loop_requested` と `auto_refresh_loop_start_requested` が出ている。
- その後、`due_channels="0"` で待機に入り、約 8 分台の `delay_ms` を経て再評価している。
- 再評価では `due_channels="81"`、次いで `45`、さらに `92`、`34` と変化しており、固定的な停止ではない。
- 最後の行は `auto_refresh_loop_sleeping delay_ms="551029"` で終わっているため、最後の睡眠後に起きたかどうかは、この断片だけではまだ確定できない。

### 検知した別件

- かなり多いのは `NSURLErrorDomain Code=-1100` のサムネイルローカルファイル欠損である。
- これは `file:///\/Library/Application Support/FeedCache/thumbnails/...` のローカル参照が存在しないことを示しており、オートリフレッシュの有無とは別に扱うべきである。
- そのほか `NSBundle (null) initWithPath failed` や AppKit の menu consistency 警告、Network / QUIC 系の `No network route` も見えるが、今回の主題を直接説明する中心ログではない。

### 手動更新の挙動

- 手動更新は短いトリガではなく、かなり大きな更新処理の前段として振る舞っていた。
- `home_auto_refresh_manual_refresh_started` の後に初回の大量更新が走り、その完了後に background loop が要求されている。
- つまり、手動更新が重いと、その後に続く自動更新の見え方も遅くなる。
- また、このトレースでは `refresh_cycle_started` に `has_manual_refresh="true"` が付いている回があり、手動更新と自動更新が同じ更新状態の上で連続していることが分かる。

## 今後の改善

- `manual refresh` をさらに `snapshot`、`dispatch`、`apply`、`ui start` などへ分割し、どこで時間を使っているかを分かるようにする。
- `sleep` の後に必ず `wake` と `snapshot` が出るかを、終端まで追えるログ長にする。
- サムネイル欠損エラーは、オートリフレッシュとは別の調査対象として分離し、件数と発生箇所を追えるようにする。
- ノイズの多い OS / Network ログは、調査対象外であることが分かる程度にだけ残し、主要なライフサイクルログを先に見えるようにする。

## Appendix B. LLM 出力の概要

### 1. 初期調査

- まず、オートリフレッシュが「起動していない」のか、「起動したが止まったように見える」のかを分けて見た。
- `home_auto_refresh_task_started` と `auto_refresh_loop_entered` の存在から、起動していない仮説は弱いと判断した。

### 2. 更新済みログの再調査

- その後の更新済みログで、`sleep -> wake -> dispatch -> cycle finish` の反復が確認できた。
- これにより、完全停止ではなく、待機と再評価を繰り返していると整理した。

### 3. 追加で見つかった別件の切り分け

- サムネイル欠損の `NSURLErrorDomain -1100` を、オートリフレッシュ本体とは別件として扱う方針にした。
- `NSBundle` や menu consistency、Network / QUIC のログは、今回の主題からは外れるノイズとして記録した。

### 4. レポート化

- 調査の収束方法、今回の例、別件、手動更新の挙動を、後で再調査できる形に再構成した。
- 冒頭に暫定結論を置き、Appendix に指示全文と要約を分けた。

## Appendix C. LLM 内での試行錯誤

### 1. 「完全停止」仮説を先に疑った

- 最初は「Mac だから止まる」「最初から動いていない」可能性を見た。
- しかし、境界ログを追うとループ起動と再開が確認でき、停止仮説は弱まった。

### 2. `sleep` 後の追跡不足を疑った

- 途中のログ断片だけだと、`sleeping` のあとに本当に戻ったか不明だった。
- そこで、`wake` と `dispatch` が続いているかを確認して、追跡不足か実停止かを切り分けた。

### 3. ノイズの多さに引っ張られないようにした

- `NSBundle`、Network、QUIC、menu consistency などの OS 由来ログは目立つが、本件の主因とは限らない。
- これらを先に深追いすると、調査が発散するため、まずライフサイクルログを優先した。

### 4. 手動更新を「遅い開始」と混同しかけた

- 手動更新の長さがあるため、最初は「更新されていない」ように見えやすい。
- しかし実際には、更新そのものが長く、完了後に背景ループが続いている。

### 5. 最終スリープ後の終端観測がまだ足りない

- 末尾の `sleeping` のあとに起床ログが入るかは、今回の断片だけでは最後まで追えていない。
- 次回の削減対象はここであり、終端までの観測を短く切れるようにログ範囲を増やすべきである。
