# Issue 4 Session Report 2026-03-29

この文書は、Issue 4 の一行目的から詳細化、実装、検証、コミット完了までを、今後の開発手順最適化に再利用できるよう整理したレポートである。主目的は LLM クォーター最適化、次点で時間最適化であり、特に「単純な入出力より、試行錯誤ループ回数がクォーター消費を指数的に悪化させる」という仮説の検証材料として残す。

## 暫定結論

- 最も効いた最適化は、実装前に Issue を「blocker 条件つき詳細化」へ固定し、曖昧点を順番に潰したことである。
- 一行目的のまま実装に入らず、`整合しないなら block` を徹底したことで、誤実装ループを避けられた。
- 実装段階では、`Issue の ToDo を 1 つずつ完了し、その都度コミットする` ルールが、差分分離と失敗範囲の局所化に効いた。
- LLM クォーター観点では、消費の大きい局面は長文生成そのものではなく、仕様未確定のまま進んだ場合に発生する往復と差分の混線である。
- 今回の残課題は、履歴追記スクリプトやテスト補助スクリプトの使い方確認に小さなループが発生した点であり、ここは次回さらに削れる。

## 目的

- 対象: Issue 4 `サムネイルキャッシュの最終アクセス時刻管理と閾値ベース廃棄を追加する`
- 開発上位目的:
  - 一行の目的から、発散せずに実装・テスト可能な粒度まで詳細化する。
  - 詳細化後は ToDo ごとに独立して実装・検証・コミットする。
  - 仕様未確定のまま試行錯誤ループへ入ることを避け、LLM クォーター消費を抑える。

## セッション全体の流れ

### 1. 一行目的から詳細化フェーズへ移した

- 最初の Issue 4 はタイトルだけで、Description もコメントもなく、背景・目的・完了条件が確定していなかった。
- この状態で実装へ進むと、仕様補完を LLM 側で行うことになり、後からの差し戻しループが増えると判断した。
- そのため、まず `整合しないなら blocker で止まる` 方針で、Issue を詳細化可能な状態へ寄せた。

### 2. 仕様を「推測禁止」で絞った

- ユーザーから段階的に与えられた情報を、コード現状と照合しながら整合判定した。
- 整合したものだけを Issue 正本へ反映し、整合しないものは blocker コメントで停止した。
- 特に効いたのは、`AsyncImage が生成された時点を参照とみなす` という定義を採用し、フレームワーク内部実装の推測へ踏み込まなかったことだった。

### 3. 実装前にコミット単位のルールを確認した

- 実装開始時に、`Issue の ToDo を 1 つこなすごとにコミットする` ルールが既存かを確認した。
- 結果として [rules-commit.md](../rules/rules-commit.md) に既に存在しており、追加は不要だった。
- この確認により、以後の作業は `ToDo 1 個 = 1 変更セット = 1 コミット` の粒度で迷わず進められた。

### 4. 実装を ToDo 単位で直列化した

- ToDo 1: サムネイル参照の最終アクセス時刻保持を追加
- ToDo 2: `AsyncImage` 生成時の参照更新を追加
- ToDo 3: 最終アクセス順の単発廃棄を追加
- ToDo 4: 下限を下回るまで継続削除を追加
- ToDo 5: 容量算出と閾値判定を cache 管理責務へ集約

### 5. 各 ToDo ごとに小さく検証した

- 各 ToDo ごとに対応する focused test を 1 本ずつ実行した。
- これにより、失敗時の確認範囲が常に直前差分だけに限定された。
- 広い回帰テストを毎回回すより、仕様固定済みの小差分に対しては、クォーターと時間の両方で有利だった。

## コスト最適化の観点から見た評価

## 仕様固定の効果

- 効果が大きかった点:
  - `整合するか` を先に問うたこと
  - `整合しないなら block して報告` を守ったこと
  - framework 内部挙動を必要以上に追わず、アプリ側で観測可能な定義へ落としたこと
- これにより、`実装したが仕様が違った` という高コストな巻き戻しを避けられた。

## コミット粒度の効果

- ToDo ごとにコミットしたことで、各変更セットの意味が明確になった。
- 実際に ToDo 4 と ToDo 5 の差分が一度混ざったが、コミット前に分割し直せた。
- もしこれを最後にまとめていたら、説明、検証、Issue 更新、差分切り分けのコストが増えていた。

## ループ削減の効果

- 大きなループは回避できた。
- 小さなループとして残ったもの:
  - 履歴追記スクリプトの引数形式確認
  - `collect-test-metrics` の利用断念と手動 `xcodebuild test` への切替
  - ToDo 4 と ToDo 5 が混ざった差分の再分割
- いずれも局所的で、仕様再解釈ループには発展しなかった。

## 暫定評価

- LLM クォーター最適化:
  - 良好。主因となる大きな試行錯誤ループは抑制できた。
- 時間最適化:
  - 良好。細かな確認はあったが、後戻りの少なさで相殺できた。
- 再現性:
  - 高い。Issue、ブランチ、コミット、focused test が対応づけられている。

## 今後の改善提案

- `Issue 詳細化テンプレート` を用意し、背景、整合条件、block 条件、完了条件を最初から埋める。
- `履歴追記スクリプト` の引数形式を短い例付きで文書化し、迷いを減らす。
- `collect-test-metrics` の simulator 選択不具合を直し、focused test 実行後の記録をワンステップ化する。
- `AsyncImage` のように framework 依存が絡む箇所は、「アプリ側で採用する定義」を先に仕様へ固定してから実装する。

## Appendix C. LLM 内での試行錯誤

この Appendix は、「今後さらに削りたいループ」を明示するための記録である。

### 1. 仕様未確定のまま進まないための停止

- Description が空の Issue に対して、最初は blocker 停止した。
- これは出力回数は増えるが、誤った実装ループを防ぐためには有効だった。

### 2. access time の整合性確認

- ユーザー認識では更新可能となっていたが、実コードには access time 列がなかった。
- ここで迎合せず blocker にしたことで、後続の実装や説明の巻き戻しを防いだ。

### 3. `AsyncImage` まわりの境界設定

- `生成されたら参照とみなす` という定義が入る前は、先読みや破棄の挙動まで踏み込むと framework 推測になりやすかった。
- ここを定義で切ってもらえたことで、探索範囲を縮められた。

### 4. ToDo 4 と ToDo 5 の差分混線

- 一度、残り 2 つの ToDo を同じ差分へ積んでしまった。
- しかしコミット前に検知し、TODO 4 のみ成立する形へ一旦戻してから、TODO 5 を積み直した。
- `1 ToDo = 1 Commit` のルール確認が、ここで実際に効いた。

### 5. 履歴追記スクリプトの引数確認

- `append-decision-history` と `append-metrics-history` の使い方で小さな試行錯誤があった。
- `--summary` ではなく、`--decision-line`、`--reason-line`、`--metric-line` が必要だった。
- さらに行頭記法にも制約があり、ここは再利用可能な手順として文書化価値が高い。

### 6. テスト補助スクリプトの利用断念

- `collect-test-metrics` は simulator 選択まわりでそのまま使えなかった。
- そのため、focused test は直接 `xcodebuild test` を実行し、計測履歴だけ別で追記した。
- これは大事故ではないが、自動化不足による微小ループだった。

## Appendix D. 今回の学び

- 一行目的から直接コードへ飛ばず、`block 条件つき詳細化` を挟むほうが、最終的な LLM クォーター消費は小さくなりやすい。
- 仕様の曖昧さを LLM が埋めるほど、後からの修正コストが大きくなる。
- 実装の粒度を ToDo 単位まで落とし、focused test とコミットを直列化すると、差分混線による再説明コストを抑えられる。
- 削るべき次の対象は、補助スクリプト利用時の小さな手順確認ループである。
