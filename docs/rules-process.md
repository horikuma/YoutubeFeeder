# Process Rules

この文書は、このリポジトリの開発フローに関する上位ルールを定める正本である。ここでは、開発単位、基本手順、テストと検証、完了条件、開発中の判断基準を扱う。

上位方針と rules コレクション全体の役割分担は [rules.md](./rules.md)、文書体系と履歴運用は [rules-document.md](./rules-document.md) を参照する。

## 開発プロセス

### 単位

- `開発シーケンス` は、1 件のユーザー指示を起点に開始し、必要な確認、実装、検証、文書同期を経て、最後のコミットで完了する一連の流れとして扱う。
- `開発セッション` は、その日の最初の処理開始から始まり、[rules-document.md](./rules-document.md) に従う日次ローテーションで前日分を確定した時点で完了する日次の運用単位として扱う。

### 基本手順

- 1 周の開発は、`ユーザー指示の理解`、`先行テストで期待固定`、`実装と健康度点検`、`検証`、`文書同期`、`コミット` の順で進める。
- 着手時は、ユーザー指示を読み取り、関連コードと関連文書を確認し、変更対象と影響範囲を把握する。
- 開発着手時の文書準備や日次ローテーションは [rules-document.md](./rules-document.md) に従う。

### テストと検証

- 機能追加では、実装前に期待する振る舞いを表すテストを先に追加し、そのテストで失敗を確認してから実装へ進む。
- 不具合対応では、想定原因候補ごとに再現または非再現を担保するテストを先に追加し、そのテストで失敗を確認してから修正へ進む。
- 実装中は、責務境界、複雑度、リソースの作用域、状態管理の健全性を同時に点検し、悪化する変更を放置しない。
- 検証では、変更内容に応じたテストと build 確認を行い、`error 0` かつ `warning 0` を確認する。
- 検証が最初の 1 回で通らなかった場合は、修正ループ中の再確認を関連テストと影響が及ぶはずのテストへ限定し、それらが通った後で最後に全体テストを実行する。
- 最終の全体テストでは、`scripts/collect_metrics.sh` を使って build 時間、全体 test 時間、起動性能、`docs/test-metrics.md` を同じ 1 回の全体実行から取得し、同じ全体テストを別スクリプトで重複実行してはならない。
- `tools`、`skills`、`scripts` だけを変更した場合は、アプリ本体の build や test は実施せず、対象ツールの構文確認と代表的な 1 経路の実行確認で検証する。

### 文書同期と完了条件

- 文書同期の対象と更新方法は [rules-document.md](./rules-document.md) に従う。
- 開発シーケンスの最後では、文書更新へ入る直前に [rules.md](./rules.md) を再度読み返し、長期スレッドやコンテキスト圧縮の後でもルール外の更新を混ぜていないことを確認する。
- ソースコード変更を含む場合は、ユーザーから明示的に止められていない限り、テスト、文書更新、計測記録を含めて最後にコミットまで実施する。
- ドキュメントのみを変更した場合も、ユーザーから明示的に止められていない限り、その開発シーケンスに必要な文書更新と当日ログ更新を行い、最後にコミットまで実施する。
- 細かいコミットが積み上がること自体は許容し、未確定の複数シーケンスを 1 つの大きなコミットへまとめることより、各開発シーケンスをコミットで確定することを優先する。
- コミット対象は、その時点で完了条件を満たした変更セット全体とし、コミットメッセージは日本語で記述する。

## 開発中の判断基準

- この節の判断基準には、実装中の局所的な書き方だけでなく、文書整理、秘密情報の扱い、健全性観測、試行回数の自己診断、しきい値超過時の停止判断まで含める。

- テストのためだけの分岐は、本番実装へ広げずテスト支援レイヤーへ閉じ込める。
- 既存文書の章構成を尊重し、関連するルールは適切な節へ整理して記述する。
- 秘密情報がチャットへ貼られた場合は、その値を以後の記録文書へ転記せず、必要ならマスクして扱う。

### テスト原則

- テスト戦略は、まず domain / logic 側で固定できる契約を `unit test` または軽量統合テストへ寄せ、画面層でしか守れない契約だけを `UI test` へ残す方針で組み立てる。

#### 役割分担

- テストは `unit test` と `UI test` の責務を分ける。
- テストは後追いの確認手段ではなく、仕様を明確化し、仕様競合や成立不能条件を早期検出するための正本の一部として扱う。
- 純粋なルール、並び順、parser、resource 読込は `unit test` で担保する。
- 画面遷移、主要導線、操作体験、モック経路は `UI test` で担保する。
- UI テストへ載せるのは、`ユーザー導線として壊れて困る契約` と `画面層でしか確認できない契約` に限定する。

#### 寄せ方

- UI の見た目、画面遷移、画面内フィードバック、操作導線を変更していない場合は、影響箇所が UI 層にあっても UI テストを省略してよく、その代わりに unit test や軽量統合テストで変更契約を固定する。
- 宣言的 UI 基盤や標準フレームワークへ委ねている振る舞いは、独自ロジックが無い限り UI テストで過剰に縛らず、まず pure logic や軽量統合テストへ寄せる。
- UI テストの所要時間は品質コストの一部として扱い、削減時は `代わりに何を unit test で固定したか` を明確にしたうえで、必要な導線テストは残す。
- 非同期ロード後の件数、内部状態の集約結果、並び順の細部、サマリー表示の内容のように UI 以外でも検証できる内容は、まず `unit test` または軽量な統合テストへ寄せる。
- UI 導線から起動される機能でも、domain / logic 側へ切り出せる判定や状態遷移がある場合は、まずその層でロジックテストを追加し、その後に UI を写像として確認する。

### 健全性の観測

- 機能追加や障害修正では、動作確認だけでなく `責務境界`、`複雑度`、`リソースの作用域`、`状態管理の健全性` を同時に点検する。
- 点検の起点として既存の集約観測を使い、少なくとも `ファイル行数`、`関数行数`、`1ファイル内の型数`、`公開状態量` を継続観測する。
- 開発プロセスの健康度として、`LLM試行回数` と `同一論点の再試行回数` を毎回自己診断する。

### 試行回数の数え方

- `LLM試行回数` は、1 つのユーザー指示に対して `仮説 -> 実装またはテスト -> 失敗 -> 修正` を何巡したかの回数として数える。
- `同一論点の再試行回数` は、同じ失敗原因に対してテストや実装の修正をやり直した回数として数える。
- TDD のために先に追加した `失敗するべき新規テスト` を、その後の実装で通した `赤 -> 緑` の 1 往復は、`LLM試行回数` と `同一論点の再試行回数` のどちらにも含めない。
- 目安として、`LLM試行回数 >= 4`、または `同一論点の再試行回数 >= 3` に達した場合は、開発速度と精度のバランスが崩れている可能性が高いとみなす。
- 自己診断がしきい値に抵触した場合は、設計上の問題かプロセス上の問題かを切り分けるまで新規実装を中止し、その旨と観測内容をユーザーへ返して次の指示を待つ。

### 健全性のしきい値

- `Swift ファイル行数` は原則 `500` 行未満を維持し、`900` 行以上は分割なしに進めてはならない。
- `関数行数` は原則 `60` 行未満を維持し、`80` 行以上は分割または責務移譲なしに進めてはならない。
- `1ファイル内の型数` が `12` を超える場合は、モデル、永続化、外部通信、UI 補助などが混在していないか見直す。
- `ObservableObject` や coordinator の `@Published` 状態は `8` 個前後を目安とし、増え続ける場合は別 service、store、または画面ローカル state へ戻すことを検討する。

### 秘密情報と外部連携

- API キーやトークンなどの秘密情報は、リポジトリへコミットせず、ローカル専用かつ ignore 済みの設定ファイルから注入する。
- 外部 API の認証情報は、可能な限り URL クエリへ載せず、ログや履歴へ残りにくい経路を優先する。
- 外部 API 用のキーは用途ごとに分け、利用 API やアプリ種別で制限できる前提で扱う。
