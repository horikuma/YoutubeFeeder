# Domain Rules

この文書を、実装言語や shell wrapper のような言語単位ルールの正本として扱わなければならない。ここへ記述してよい内容は、プロダクト固有の責務分割や画面仕様ではなく、組織横断で再利用する原理原則だけに限定し、それ以外を混在させてはならない。

rules コレクション全体の役割分担を判断する場合に限って [rules.md](../rules.md) を参照しなければならず、開発フローを判断する場合に限って [rules-process.md](./rules-process.md) を参照しなければならない。`tools`、`skills`、`scripts` の配置規則を判断する場合に限って [rules-skills.md](./rules-skills.md) を参照しなければならず、それ以外の論点ではこれらを参照してはならない。

## この文書の位置付け

- `rules-domain.md` では、shell、Python、C 系言語の実装原則、formatter / lint、複雑度しきい値のような言語単位ルールだけを定義しなければならない。これに該当しない内容を記述してはならない。
- この文書へ、特定プロダクトの責務分割、画面仕様、データフロー、個別ディレクトリ構成を持ち込んではならない。
- プロダクトコード固有の責務分割を更新する場合は `specs` 側へ記述しなければならず、言語をまたいで再利用できない制約を本書へ記述してはならない。
- `skills` の内部整理を扱う場合でも、プロダクトコードの内部設計と同一視してはならない。探索の幅を残す必要があるため、`skills` に適用する制約は product code より緩い別枠として定義しなければならない。
- 言語単位のルールだけを更新し、プロダクト仕様が変わらない場合は、`specs` を更新対象へ広げてはならない。

## 配置判断

- shell wrapper の責務、分岐禁止、環境変数やパス解決を定義する場合は本書へ記述しなければならない。これらを他の rules 文書へ分散させてはならない。
- Python や C 系言語の複雑度しきい値、formatter / lint のような実装言語ルールを定義する場合は本書へ記述しなければならない。
- `skills` をどの粒度で分割するかのような、言語横断で再利用する制約を定義する場合は本書へ記述しなければならない。
- `tools`、`skills`、`scripts` のディレクトリ責務や配置規則を定義する場合は [rules-skills.md](./rules-skills.md) へ記述しなければならず、本書へ書いてはならない。
- プロダクトコードの責務分割、データフロー、レイヤ境界を定義する場合は `specs` 側へ記述しなければならず、本書へ書いてはならない。

## shell ルール

- Mac で shell を扱う場合は `zsh` を標準 shell として扱わなければならず、Linux では `bash`、Windows では `PowerShell` を標準 shell として扱わなければならない。これに該当しない既定 shell を前提にしてはならない。
- shell script は、パス解決、環境変数受け渡し、実行系の起動だけを担う薄い wrapper としなければならない。これらに該当しない実装本体を持ち込んではならない。
- shell script に業務ロジック、状態管理、`if` / `case` / `while` / `for` を使った分岐処理を持ち込まない。
- 条件分岐や複雑な判定が必要になった時点で、shell に留めず Python の実装本体へ移す。
- `zsh` と `bash` をまたぐ wrapper は、`bash` で作成しても `bash` 専用構文へ寄せすぎず、Mac / Linux の互換性を優先する。
- `zsh` と `bash` をまたぐ wrapper では、配列依存、`[[ ... ]]` の多用、process substitution のような互換性を崩しやすい記法を避け、POSIX に近い書き方を優先する。
- wrapper は引数をそのまま実装本体へ透過し、引数の意味変換や暗黙補完を増やさない。
- shell を追加または変更した時は、少なくとも `bash -n` で構文確認する。

## Python ルール

- shell wrapper の実体は、特別な制約がない限り Python に置かなければならない。理由なく他言語へ分散させてはならない。
- Python の共有実行環境はリポジトリ root の `.venv/` を正本として扱わなければならず、共有依存定義はリポジトリ root の `requirements.txt` へ集約しなければならない。共有依存を局所定義へ分散させてはならない。
- 依存が増えた場合は、局所 requirements を増やす前に root の `requirements.txt` へ集約できないかを先に検討する。
- Python の構文確認と lint の最低ラインは `py_compile` とし、変更した Python ファイルに対して `python -m py_compile` を実行して確認する。
- Python の product code は、1 ファイル `500` 行未満を原則とし、`800` 行以上は分割なしに進めてはならない。
- Python の product code は、1 関数 `60` 行未満を原則とし、`90` 行以上は分割または責務移譲なしに進めてはならない。
- Python の product code は、1 ファイル内の主要型や主要責務が `10` を超える場合、責務の混在を疑って分割を検討する。

## C 系言語ルール

- C 系言語の formatter は `clang-format` を正本として扱わなければならず、リポジトリ root の `.clang-format` を適用しなければならない。別の formatter を既定として扱ってはならない。
- `clang-format` は、C、C++、header を含む実装語全体へ適用しなければならない。対象言語の一部だけを恣意的に除外してはならない。
- C 系 product code は、1 ファイル `800` 行未満を原則とし、`1200` 行以上は分割なしに進めてはならない。
- C 系 product code は、1 関数 `80` 行未満を原則とし、`120` 行以上は分割または責務移譲なしに進めてはならない。
- C 系 product code は、1 ファイル内の主要型や主要責務が `12` を超える場合、責務の混在を疑って見直す。

## skills の実装制約

- `skills` を分割する場合は、プロダクトコードと同じ内部レイヤ分割を無理に模倣してはならず、公開する skill や command の機能単位で分割しなければならない。
- `skills` では、探索や試行の余地を残すため、プロダクトコードより緩い複雑度しきい値を使わなければならない。product code と同一しきい値を機械的に適用してはならない。
- `skills` の複雑度が上がった場合は、まず内部構造を細分化するより先に、skill 単位または command 単位へ分割できないかを確認する。
- `skills` の Python 実装は、1 ファイル `800` 行未満を原則とし、`1200` 行以上は分割なしに進めてはならない。
- `skills` の Python 実装は、1 関数 `100` 行未満を原則とし、`140` 行以上は分割または補助関数への抽出なしに進めてはならない。
- `skills` の shell wrapper は product code と同様に薄い wrapper とし、分岐や実装本体を持ち込まない。
- `skills` の分割判断は `specs` の内部責務ではなく、利用者に公開する機能境界と再利用単位を基準に行う。

## 更新判断

- 実装言語ごとの formatter / lint / 複雑度しきい値を変えた時は、本書を更新しなければならない。更新せずに実装だけを変えてはならない。
- shell wrapper の責務や互換性方針を変えた時は、本書を更新しなければならない。
- `skills` の複雑度や分割方針を変えた時は、本書と [rules-skills.md](./rules-skills.md) の責務分担を確認しなければならない。どちらに記述すべきかを曖昧なまま変更してはならない。
