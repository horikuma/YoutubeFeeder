# Report Creation Rules

この文書は、レポート作成・更新タスクを単体で定義する完結文書である。

## レポート作成・更新

- レポート作成・更新とは、`docs/report/` 配下に再利用可能な報告文書を新規作成または更新し、後続の判断や改善に使える状態へ整えるタスクである。

## 実施内容

- レポートは、対象タスク、対象日付、目的がタイトルまたは冒頭で識別できる文書として作成しなければならない。
- レポートのファイル名は、`YYYY-MM-DD-<対象または目的>.md` の形式で、日付を先頭に置かなければならない。
- レポート冒頭には、最初に読むべき結論として `暫定結論` または同等の結論節を配置しなければならない。
- レポート冒頭の結論には、何が最も効いたか、何が主な課題として残ったか、次回どこを削減または改善すべきかを含めなければならない。
- レポート本文には、少なくとも `目的`、`実施の流れまたは工程`、`評価`、`今後の改善` に相当する節を含めなければならない。
- レポートがコスト最適化や効率化を目的とする場合は、何のコストを対象としたかを明示し、主目的と副目的の優先順位を本文に書かなければならない。
- レポートが LLM 利用を扱う場合は、単純な入出力ではなく、どの試行錯誤やループがコスト増大に効いたかを本文または Appendix に明示しなければならない。
- レポートには Appendix を設け、少なくとも次の 3 点を分離して残さなければならない。
  - ユーザー指示全文
  - LLM 出力の概要
  - LLM 内での試行錯誤、または今後削減したいループ
- Appendix に記載するユーザー指示全文は、判断に影響した指示を順序を保って再掲しなければならない。
- LLM 出力の概要は、詳細化、実装、検証、報告などの主要フェーズ単位で要約しなければならない。
- 試行錯誤の記録は、何がループの原因だったか、どう収束したか、次回どう削るかを読める形で残さなければならない。
- レポートは、文書だけ読めば判断の背景と改善余地が追える構成にしなければならない。
- レポート作成後は、必要な `docs/history/*-latest.md` 更新を行い、文書変更として追跡可能な状態へ整えなければならない。

## 完了条件

- `docs/report/` 配下に、対象、日付、目的が識別できるレポートが存在すること。
- レポートファイル名が、`YYYY-MM-DD-<対象または目的>.md` の形式で日付先頭になっていること。
- レポート冒頭に暫定結論があり、本文に目的、工程、評価、今後の改善が含まれていること。
- Appendix にユーザー指示全文、LLM 出力概要、試行錯誤の記録が分離されていること。
- レポート単体を読めば、対象タスクの流れ、成果、失敗やループ、次回改善点が把握できること。
- 文書変更に対応する `docs/history/*-latest.md` 更新が完了していること。

## 禁止事項

- 結論を末尾まで遅らせ、冒頭で重要判断が分からない構成にしてはならない。
- 目的を書かずに、経緯や感想だけを列挙してはならない。
- コスト最適化を扱うレポートで、対象コストや優先順位を曖昧にしてはならない。
- ユーザー指示全文を省略したり、判断順序が分からない形で断片化してはならない。
- LLM 出力の概要と試行錯誤を混同し、何が成果で何が削減対象か分からない構成にしてはならない。
- 試行錯誤の記録を、次回改善に使えない抽象的な反省だけで終わらせてはならない。
- 文書外の会話や記憶を知らなければ理解できない前提で、省略したレポートを書いてはならない。
