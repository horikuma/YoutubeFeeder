# Rule Creation Rules

この文書は、ルール作成・更新タスクを単体で定義する完結文書である。

## ルール作成・更新

- ルール作成・更新とは、`docs/rules/` 配下のルール文書を新規作成または更新し、正本として利用可能な状態へ整えるタスクである。
- ルール文書で command を記述する場合は、Issue作成ルールと同様に usage 記法で記述しなければならない。
- ルール文書で使う usage 記法では、角括弧 `[...]` 内は省略可能部分を表す。
- ルール文書で使う usage 記法では、角括弧の外にある要素は必須であり、左から右の順にそのまま指定しなければならない。
- ルール文書で使う usage 記法では、山括弧 `<...>` 内は実行時に具体値へ置換して渡す値を表す。

## 実施内容

- ルール文書は 1 ファイルで完結する内容として記述しなければならない。
- 対象ルールの判断、完了条件、実施内容、禁止事項は、そのルール文書単体で理解できるように書かなければならない。
- ルール文書には、誰が何をいつ行うかを曖昧語なしで記述しなければならない。
- ルール文書へ command の規定を追加する場合は、実施内容の箇条書きの直下に command の usage を 1 行で記述し、その下へ各置換値、生成物、分岐条件を 1 項目ずつ列挙しなければならない。
- ルール文書へ `skills/` 配下の skill またはその command を使う規定を追加する場合は、どの skill / command を、どの引数または入力で、何のために使うかを usage と箇条書きだけで具体的に記述しなければならない。
- 条件分岐がある場合は、実行条件と非実行条件を明示しなければならない。
- 禁止事項がある場合は、禁止対象と禁止範囲を明示しなければならない。
- 完了条件がある場合は、満たすべき状態を観測可能な形で明示しなければならない。

## 完了条件

- 作成または更新したルール文書が、単体で読んで判断できる内容になっていること。
- 作成または更新したルール文書が、他のルールファイルを参照せずに成立していること。
- 作成または更新したルール文書が、曖昧な補完なしに一意に読める内容になっていること。
- command を含む規定がある場合は、usage、置換値、生成物、分岐条件がそのルール文書内に記述されていること。
- `skills/` 配下の skill またはその command を使う規定を含む場合は、具体的な使い方まで記述されていること。

## 禁止事項

- ルール文書を、他のルールファイルを読まなければ成立しない構成にしてはならない。
- ルール文書の判断、完了条件、実施内容、禁止事項を、他のルールファイルへの参照で代替してはならない。
- command 名だけを書き、usage、置換値、生成物、分岐条件を省略してはならない。
- skill 名や command 名だけを書き、具体的な使い方を文書外の実装読解へ委ねてはならない。
- 常識や慣習で補うことを前提にした記述をしてはならない。
- 曖昧語、例示依存、暗黙の前提だけで判断させてはならない。
- 文書外の意図を読めば分かるという前提で、条件や禁止事項を省略してはならない。
