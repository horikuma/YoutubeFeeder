# Sequence End Rules

この文書は、開発シーケンス終了タスクを単体で定義する完結文書である。

## 開発シーケンス終了

- 開発シーケンス終了とは、実装結果を検証し、必要な文書更新と記録を終え、変更を履歴と Pull Request へ確定するタスクである。

## 実施内容

- 変更内容に応じた検証を実施し、必要な build 確認とテストを完了させる。
- 検証が最初の 1 回で通らなかった場合は、修正ループ中の再確認を関連範囲へ限定し、その後に最終の全体確認を行う。
- `tools`、`skills`、`scripts` だけを変更した場合は、アプリ本体の build や test ではなく、対象ツールの構文確認と代表的な 1 経路の実行確認で検証する。
- 必要な文書同期を完了させる。
- 文書更新へ入る直前に、rules の索引を再確認し、役割外の更新が混ざっていないことを確認する。
- ソースコード変更を含む場合は、ユーザーから明示的に止められていない限り、テスト、文書更新、計測記録を含めてコミットする。
- ドキュメントのみを変更した場合も、ユーザーから明示的に止められていない限り、必要な文書更新と当日ログ更新を含めてコミットする。
- コミット対象は、その時点で完了条件を満たした変更セット全体とし、コミットメッセージは日本語で記述する。
- 関連する検証と文書同期を終えた後、必ず Pull Request を作成する。
- Pull Request には Issue と同じ解決経路で Assignee を設定する。
- Pull Request を Project へ自動登録してはならない。
- LLM は Issue を直接 close してはならない。
- GitHub Project に `LLM所要時間` フィールドがある場合は、開発シーケンス終盤で実測した分数を反映する。
- Issue 駆動で進めたタスクは、Issue、ブランチ、コミット、Pull Request の対応関係が追跡できる状態で終える。

## 完了条件

- 必要な検証が完了していること。
- 必要な文書同期と記録が完了していること。
- 変更セットがコミットで確定していること。
- 対応する Pull Request が作成済みであること。

## 禁止事項

- 検証未完了のまま、開発シーケンスを完了扱いにしてはならない。
- 文書同期や記録を飛ばして、コミットや Pull Request だけを先に確定してはならない。
- Pull Request を作らずに、開発シーケンスを完了扱いにしてはならない。
- LLM が Issue を直接 close してはならない。
