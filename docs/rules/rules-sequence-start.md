# Sequence Start Rules

この文書は、開発シーケンス開始タスクを単体で定義する完結文書である。

## 開発シーケンス開始

- 開発シーケンス開始とは、1 件のユーザー指示を、その後の実装へ進められる開発単位へ立ち上げるタスクである。
- このタスクは、開発セッション開始タスクの完了後に実施する。

## 実施内容

- ユーザー指示を読み取り、関連コードと関連文書を確認し、変更対象と影響範囲を把握する。
- 現在のシーケンスで使う session main と GitHub 操作モードを取り違えないように確定する。
- 開発セッション情報が未確定なら、現在のブランチをこのセッションの基準ブランチとして使うかをユーザーへ確認する。
- ユーザーが現在のブランチを使うと答えた場合は、そのブランチをセッション限定の main として扱う。
- ユーザーが現在のブランチを使わない場合は、通常の `main` をこのシーケンスの基準ブランチとして扱う。
- 開発セッション情報がすでに確定している場合は、同じ確認を繰り返さず、既存の開発セッション情報を参照する。
- チャット欄からの指示で開発シーケンスを始める場合は、実装へ進む前に、その指示を原文として残した Issue を作成する。
- Issue を作成した後は、そのシーケンスを Issue 駆動の開発として扱う。

## 完了条件

- 変更対象と影響範囲を説明できる状態になっていること。
- そのシーケンスで使う session main と GitHub 操作モードが確定していること。
- チャット起点の指示なら、原文を保持した Issue が作成済みであること。

## 禁止事項

- session main と GitHub 操作モードが未確定のまま、以後の工程へ進んではならない。
- 同じ開発セッション内で、すでに確定している session main の確認を繰り返してはならない。
- チャット起点タスクで、Issue を作らずに実装へ進んではならない。
