# HelloWorld App Specification

この文書は、HelloWorld アプリの要求仕様と機能仕様を記述する正本である。人間および LLM が、このアプリが何をしなければならないか、どのように振る舞わなければならないか、何を壊してはいけないかを同じ基準で理解するために用いる。

文書群全体での役割分担は [document-roles.md](./document-roles.md)、文書運用ルールは [document-operations.md](./document-operations.md) を参照する。機能を変える場合は、関連する正本と人間向け参照資料に矛盾が出ないよう更新する。

## 1. 目的

このアプリは、登録済みの YouTube チャンネル群の feed を取得し、ローカルキャッシュとして保持し、ユーザーがホーム画面、チャンネル一覧、動画一覧から最新動画を快適に確認できるようにする iOS アプリである。

このアプリが重視することは次の通り。

- 起動直後に最低限の画面を速く表示すること
- feed 取得と UI 表示を分離し、操作中に固まりにくいこと
- 取得済みデータを再利用し、毎回フル取得しないこと
- 将来の並び替え、検索、フィルタ追加に耐えるデータ構造であること
- 人と LLM の両方が、同じ仕様文書から変更判断できること
- 起動や検証に関する客観的な観測値を蓄積し、性能劣化を早期に把握できること

## 2. 対象プラットフォーム

- iOS アプリである
- `iPhone` を主設計対象とする
- `iPad` でも同一機能を提供する
- GUI は `共有する振る舞い` と `端末や向きで分ける振る舞い` を区別して定義する

## 3. 用語

- `ホーム画面`: アプリの主画面。起動後に最初に入る画面
- `軽量キャッシュ` または `bootstrap`: ホーム画面を最短で表示するための小さいキャッシュ
- `本体キャッシュ`: チャンネル状態、動画メタデータ、サムネイル保存先を含む主キャッシュ
- `更新確認`: feed 本体を取る前に、feed が更新されているかだけを調べる処理
- `本体取得`: 更新ありと判定されたチャンネルの feed 本体を取得する処理
- `新着動画`: その時点の本体キャッシュにまだ存在しなかった動画

## 4. 画面構成

現時点のユーザー向け画面は次の通り。

1. 起動画面
2. ホーム画面
3. チャンネル登録画面
4. チャンネル一覧画面
5. チャンネル別動画一覧画面
6. 動画一覧画面

## 5. 機能要件

### 5.1 起動

- アプリ起動時は最初に `LaunchScreenView` を表示する
- 起動直後は重い処理を行わず、まず軽量キャッシュのみを読み込む
- 軽量キャッシュの読込が完了したらホーム画面へ遷移する
- 起動時に本体キャッシュの全読込や feed 更新を自動開始してはならない

### 5.2 ホーム画面

ホーム画面は次を表示する。

- `チャンネル` への導線
- `動画` への導線
- `キャッシュ検索` への導線
- `YouTube検索` への導線
- `チャンネル登録` への導線
- `バックアップ` への導線
- `全設定リセット` への導線
- `システム状況` 情報タイル

ホーム画面は次の操作を提供する。

- 下方向の pull-to-refresh による手動更新
- `チャンネル` タイルから並び順を選んでチャンネル一覧画面への遷移
- `動画` タイルから動画一覧画面への遷移
- `キャッシュ検索` タイルから固定キーワード検索結果画面への遷移
- `YouTube検索` タイルから YouTube 検索結果画面への遷移
- `チャンネル登録` タイルからチャンネル登録画面への遷移
- `バックアップ` タイルからこの端末内バックアップの書き出し / 読み戻し
- `全設定リセット` タイルから、この端末内のチャンネル設定・動画キャッシュ・検索履歴・サムネイルを削除
- `システム状況` は非操作タイルとして表示し、押下導線と誤認しにくい見た目にする

### 5.3 チャンネル登録

- ホーム画面から `チャンネル登録` を開けること
- 入力は `Channel ID`、`@handle`、YouTube のチャンネル URL、動画 URL を受け付けること
- 動画 URL を入力した場合は、その動画の投稿元チャンネルを登録すること
- `@handle` や URL を入力した場合は、登録前に対応する `Channel ID` へ解決すること
- 永続化時の主キーは、入力文字列ではなく解決後の `Channel ID` とすること
- すでに存在する `Channel ID` は重複登録しないこと
- 追加したチャンネルは、その後のチャンネル一覧および更新対象に含まれること
- 登録結果として、新規登録か既登録かを画面上で明示すること
- 登録結果として、解決後のチャンネル名、`Channel ID`、最新動画、公開日、保持動画件数を可能な範囲で画面上へ返すこと
- 最新動画の取得に失敗した場合でも、登録自体の成否と失敗理由を区別して表示すること

### 5.4 バックアップ

- ホーム画面から、チャンネル設定をこの端末内の固定ファイルへ書き出せること
- ホーム画面から、同じ固定ファイルを読み込んでチャンネル設定を復元できること
- 本機能は他デバイス間移動ではなく、1 デバイス内でのバックアップ / 復元を目的とすること
- ファイル形式は将来拡張に耐える JSON とすること
- 書き出し対象はチャンネル設定に限り、サムネイルや動画キャッシュ本体は含めないこと
- 現在登録されている全チャンネル情報を保存対象とすること
- 読み込み自体はホーム画面の処理中表示を長時間引き延ばさずに完了すること
- 読み込み後は、その端末側で最新動画情報の再取得をバックグラウンドで開始すること
- 固定ファイルパスは `~/Documents/HelloWorld/channel-registry.json` 相当の `Documents/HelloWorld/channel-registry.json` とすること
- バックアップファイルが存在しない場合は、ホーム画面上で明確に失敗理由を返すこと

### 5.5 全設定リセット

- ホーム画面から、この端末内の設定とキャッシュをまとめて削除できること
- 削除対象は channel registry、動画キャッシュ、bootstrap、検索履歴、サムネイルとすること
- `Documents/HelloWorld/channel-registry.json` のバックアップファイルは削除対象に含めないこと
- 実行前に確認 UI を出し、破壊的操作であることを明示すること
- 実行後は、何件のチャンネル / 動画 / サムネイル / 検索履歴を削除したかを画面上で返すこと
- リセット後でも、ホーム画面からバックアップを読み込んで復旧できること

### 5.6 チャンネル一覧画面

- `チャンネル` タイルから遷移する
- ホーム画面の `チャンネル` 導線では、`指標` と `昇順 / 降順` を選んでから一覧を開けること
- 初期選択は `動画投稿日時` の `降順` とすること
- 現時点では `動画投稿日時` と `チャンネル登録日時` を指標として選べること
- キャッシュ済みチャンネルは、選択した並び順に従って表示すること
- チャンネル一覧の先頭には、登録件数、現在の並び順、基本操作をまとめた非操作の `Tips` タイルを表示すること
- 1 チャンネルを選ぶと、そのチャンネルの動画一覧へ進める
- Adaptive UI により分割レイアウトが成立する環境では、チャンネル一覧と選択中チャンネルの動画一覧を同時表示してよい
- 分割レイアウトでは、チャンネルを切り替えても内容表示側だけが更新されること
- チャンネルタイルの長押しでは、削除メニューを開けること

理由:
- 広い表示幅が使える環境では、チャンネルを切り替えながら内容を比較できることの利便性が高い。
- 一方で分割レイアウトが成立しない環境では、単独画面遷移のほうが理解しやすく、Adaptive UI の既定挙動にも沿う。

### 5.7 チャンネル別動画一覧画面

#### 基本表示

- 選択した 1 チャンネルのキャッシュ済み動画を新しい順に最大 50 件表示する
- Shorts は表示しない
- 単独画面レイアウトでは独立画面として表示する
- 分割レイアウトでは一覧側の選択に対応する内容表示領域として表示する

#### 更新と自動読込

- この画面で下方向に pull-to-refresh した場合は、一覧全体ではなく選択中の 1 チャンネルだけを強制更新すること
- YouTube検索結果の動画タイルから遷移した場合、初回表示でそのチャンネルのローカル feed キャッシュが未作成、または選択した動画がローカル channel cache に存在しない時は、自動的に当該チャンネルの feed を強制更新すること
- 上記の自動 feed 更新が走っている間は、pull-to-refresh 時の上部スピナーと同系統の UI で、読み込み中であることをユーザーへ明示すること
- YouTube検索結果から遷移した場合は、検索時に取得済みのチャンネル名を初回表示タイトルへ即座に反映し、feed 更新完了を待たずにチャンネル名が見えること
- チャンネル別動画一覧は feed 更新のたびに置き換えず、既存キャッシュへ append / merge すること

#### 動画タイル

- この画面の動画タイルには、画面内の通し番号を 1 から表示すること
- 動画タイル右上には、チャンネル一覧画面の番号バッジと同じ書式で通し番号を表示すること
- 動画タイル右下には、`丸めた分数 + 再生数` の形式で情報を表示し、再生時間区分の `(M)/(L)` は表示しないこと
- この画面でだけ、動画タイル長押しメニューから `YouTubeで開く` を選べること

#### 診断

- 実機調査用ログを有効にした起動では、下方向の pull-to-refresh からフィード取得、整合性メンテナンス、一覧反映までの経路を時系列ログとして出力できること

### 5.8 動画一覧画面

- キャッシュ済み動画を新しい順に最大 50 件表示する
- Shorts は表示しない
- 見た目は大きいヒーロータイルとする
- 縦スクロール可能であること
- 表示中はキャッシュ更新の live 反映でスクロールや操作感を阻害しないこと
- 動画タイルの通常タップでは、その動画の属するチャンネル別動画一覧へ遷移すること
- 動画タイルの長押しでは、`チャンネルを削除` を選べること
- 動画タイルには、画面内の通し番号を 1 から表示すること
- 動画タイル右上には、チャンネル一覧画面の番号バッジと同じ書式で通し番号を表示すること
- 動画タイル右下には、`丸めた分数 + 再生数` の形式で情報を表示し、再生時間区分の `(M)/(L)` は表示しないこと

### 5.9 固定キーワード検索結果画面

- ホーム画面の `キャッシュ検索` タイルから遷移すること
- キーワードは固定で `ゆっくり実況` とすること
- 現在のキャッシュから一致する動画を新しい順に最大 20 件表示すること
- Shorts は表示しないこと
- 画面下部に、検索結果が何件あったかを示す一時的なチップを表示すること
- チップは自動タイマーでは消さず、スクロール、タップ、更新などユーザー操作が始まるまで表示を維持すること
- 動画タイルの通常タップでは、その動画の属するチャンネル別動画一覧へ遷移すること
- 動画タイルには、画面内の通し番号を 1 から表示すること
- 動画タイル右上には、チャンネル一覧画面の番号バッジと同じ書式で通し番号を表示すること
- 動画タイル右下には、`丸めた分数 + 再生数` の形式で情報を表示し、再生時間区分の `(M)/(L)` は表示しないこと

### 5.10 YouTube検索結果画面

#### 基本表示

- ホーム画面の `YouTube検索` タイルから遷移すること
- キーワードは固定で `ゆっくり実況` とすること
- 画面表示時に自動検索してはならず、ユーザーが画面内で pull-to-refresh した時だけ YouTube 検索 API を実行すること
- 画面表示時は、前回の検索キャッシュがあればそれだけを表示し、無ければ未取得状態を表示すること
- YouTube 検索 API の結果から、新しい順に最大 100 件表示すること

#### 検索とキャッシュ

- 1 回の検索では `videoDuration=medium` と `videoDuration=long` の 2 検索を行い、結果を統合すること
- 2 検索の結果は `videos.list` で補完し、ライブ動画を除外したうえで 1 回分の検索結果として保存すること
- `videos.list` は動画 ID を 50 件単位でまとめて問い合わせ、50 件ヒット時に 50 回コールしてはならないこと
- `videos.list` の詳細取得では `statistics` を含め、再生数を検索結果タイルへ反映できること
- 検索結果は端末内キャッシュへ保存し、短時間の再訪で毎回 API 再取得しないこと
- 検索キャッシュの寿命は、通常の画面再訪では再取得が過剰にならない程度に長めに取ること
- pull-to-refresh 時のみ、明示的な再取得を許可すること
- API キー未設定や検索失敗時は、空画面のままにせず理由を画面上へ返すこと
- 検索履歴はキーワードごとに過去結果へ順次マージし、任意タイミングでクリアできること

#### フィードバックと表示制御

- 画面下部に、最終更新時刻、検索結果件数、検索元を示す一時的なチップを表示すること
- pull-to-refresh による再検索中は、前回検索結果の要約をそのまま残さず、上部に進行中表示を出し、下部チップは古い要約を見せないこと
- チップは自動タイマーでは消さず、スクロール、タップ、更新などユーザー操作が始まるまで表示を維持すること
- 検索結果は 1 回に全件描画せず、初回 20 件を表示し、下端到達で続きを追加読込すること

#### 遷移とレイアウト

- 単独画面レイアウトでは、動画タイルの通常タップでその動画の属するチャンネル別動画一覧へ遷移すること
- 分割レイアウトでは、検索結果動画一覧と選択中チャンネルの動画一覧を同時表示し、動画タイルの通常タップでは内容表示側だけを更新すること
- YouTube検索結果の動画タイルからチャンネル別動画一覧へ遷移した場合は、検索結果に含まれていたチャンネル名を初期表示へ引き継ぐこと
- 動画タイルには、画面内の通し番号を 1 から表示すること
- 動画タイル右上には、チャンネル一覧画面の番号バッジと同じ書式で通し番号を表示すること
- 動画タイル右下には、`丸めた分数 + 再生数` の形式で情報を表示し、検索で拾われた再生時間区分 `(M)/(L)` は表示しないこと
- 広い画面でも、動画タイルは 1 列表示を維持し、本文幅だけを読みやすい範囲へ制限すること

### 5.11 動画の再生

- 動画タイルの通常タップで動画を開いてはならない
- `YouTubeで開く` はチャンネル別動画一覧の動画タイル長押しメニューからのみ選べること
- 可能なら `youtube://watch?v=<videoID>` を開く
- YouTube アプリが開けない場合は Web URL にフォールバックする

### 5.12 チャンネル削除と整合性メンテナンス

- チャンネル削除はチャンネルタイルまたは動画タイルの長押しメニューから行えること
- 削除時は確認ダイアログを表示すること
- 削除後は、そのチャンネルに属する動画キャッシュと不要サムネイルを整理すること
- 整合性メンテナンスは、チャンネル削除時には即時、通常の更新フローでは軽い定期処理として自然に動くこと
- 整合性メンテナンスは、参照の切れたチャンネル状態、動画、サムネイルを残さないこと
### 5.13 キャッシュ対象

キャッシュは次を保持する。

#### チャンネル単位

- チャンネル ID
- チャンネル名
- 最終取得時刻
- 最終確認時刻
- 最新動画投稿日
- 動画件数
- エラー状態
- `ETag`
- `Last-Modified`

#### 動画と検索

- 各動画の ID、タイトル、チャンネル名、投稿日、動画 URL、サムネイル URL、ローカルサムネイル名、検索用文字列
- 各動画の再生時間秒数と再生数
- 固定キーワード `ゆっくり実況` に対する YouTube 検索結果キャッシュ

### 5.14 キャッシュの永続化

- キャッシュはアプリ終了後も再利用可能であること
- bootstrap と本体キャッシュは別ファイルに保存すること
- チャンネル一覧は、アプリ終了後も再利用可能であること
- サムネイルはローカルファイルとして保存すること
- アプリ再起動時は bootstrap を優先して使うこと
- YouTube 検索結果キャッシュは本体キャッシュとは別ファイルで保持すること

### 5.14 診断と観測

- 実機での再現調査時は、アプリを通常操作しながらランタイムログをリアルタイムに取得できること
- ランタイムログは、少なくとも `チャンネル別動画一覧の更新ジェスチャー`、`単独チャンネル更新の開始/完了`、`フィード取得結果`、`整合性メンテナンス結果`、`UI 反映または保留` を含むこと
- ランタイムログは、物理 `iPhone 12 mini` に対するコンソール接続から取得できること

### 5.15 手動更新フロー

手動更新はホーム画面の pull-to-refresh を起点とし、次の規則で動作する。

1. チャンネル一覧を優先順に並べる
2. 最大 3 チャンネルまで同時に処理する
3. 各チャンネルは次の順に処理する
   1. 更新確認
   2. 更新なしならそのチャンネルは終了
   3. 更新ありなら本体取得
   4. 本体取得で見つかった新着動画のうち、まだサムネイルを持たないものだけサムネイル取得
   5. そのチャンネルは終了
4. 1 チャンネルが終了したら、残りの待ちチャンネルから次を投入する

この結果、同時ネットワークリクエスト数は常に最大 3 を超えてはならない。

### 5.16 チャンネル優先順

チャンネル処理順は、`更新日時降順` とする。現在の実装上の意味は次の通り。

1. `latestPublishedAt` が新しいチャンネルを優先
2. 同じ場合は `lastSuccessAt` が新しいチャンネルを優先
3. さらに同じ場合は `lastCheckedAt` が古いチャンネルを優先

このルールは、「最近更新されたチャンネルほど、今後も更新頻度が高い可能性がある」という仮説に基づく。

### 5.17 更新確認の条件付き取得

- 更新確認では `ETag` と `Last-Modified` を使用する
- `304 Not Modified` を返せる場合は、本体取得を行わない
- 条件付き取得が効かない場合でも、通常取得へフォールバックして壊れてはならない

### 5.18 新着動画とサムネイル取得

- サムネイル取得対象は、その回の本体取得で新しく見つかった動画だけとする
- 既に本体キャッシュに存在する動画のサムネイルは再取得しない
- 既にローカル保存済みのサムネイルは再ダウンロードしない

### 5.19 一覧表示中の更新抑制

- 一覧画面表示中は `@Published` による live update を止める
- バックグラウンドでキャッシュが更新されても、その場で一覧を再描画しない
- 一覧画面から戻った時点で最新状態を再読込する

### 5.20 非機能性能の観測

- コミット単位で、build 時間、test 時間、再試行回数を保存すること
- コミット単位で、`起動からスプラッシュ表示まで`、`スプラッシュ表示からホーム表示まで`、`起動からホーム表示まで` を保存すること
- 上記の観測値は、可能な限り `xcodebuild` と UI テストから再取得可能な方法で記録すること
- 観測値の当日更新は `history/metrics-latest.md` に保存し、履歴は `history/metrics-log.md` から参照できること

## 6. UI 仕様

### 6.1 GUI 共有方針

- GUI 仕様は `共有する振る舞い` と `Adaptive UI により分かれる振る舞い` を区別して記載する
- 機能ロジックは共有し、差分は原則としてレイアウトと遷移表現に限定する
- `iPhone` と `iPad` は同一機能を提供し、分割レイアウトの採否は OS とレイアウト条件に委ねる

### 6.2 共通

- 背景は `systemGroupedBackground`
- 一覧タイルは大きいヒーロータイルを維持する
- サムネイル未取得時もレイアウトを崩さない
- 動画一覧画面は端末差分を持たず、全端末で単独画面のままとする

### 6.3 単独画面 GUI

- ホーム画面から `チャンネル` を開くと、チャンネル一覧画面へ遷移する
- チャンネル一覧でチャンネルを開くと、チャンネル別動画一覧画面へ遷移する
- 一覧画面は 1 画面 1 主題の構成とし、戻る操作で前画面へ戻る

### 6.4 分割 GUI

- ホーム画面から `チャンネル` を開いた時は、Adaptive UI により必要に応じて左右分割のチャンネル閲覧 UI を使う
- ホーム画面から `YouTube検索` を開いた時も、Adaptive UI により必要に応じて左右分割 UI を使う
- 左半分はチャンネル一覧、右半分は選択中チャンネルの動画一覧とする
- `YouTube検索` の左半分は検索結果動画一覧、右半分は選択中チャンネルの動画一覧とする
- 初期表示時は、左側の先頭チャンネルを選択状態にする
- 左側でチャンネルを切り替えると、右側の動画一覧だけを更新する
- `動画` 画面は分割レイアウト時でも単独画面のままとする

理由:
- 広い表示幅ではチャンネルの切り替えと内容確認を素早く繰り返す体験が有効であり、画面を往復するより比較しやすい。
- `動画` 画面は全体の新着を連続的に見る用途が中心であり、現状は分割 UI にする必然が低い。

### 6.5 起動画面

- `HelloWorld` の文字を表示する
- 画面は極力軽く、起動原因切り分けの基準点として使えること

### 6.6 ホーム画面

- 先頭に `ホーム`
- その下に、現在の並び順を表示した `チャンネル` タイル
- その下に `動画` タイル
- その下に `チャンネル登録` タイル
- その下に `バックアップ` タイル
- 書き出し / 読み戻しの結果や失敗理由は、ホーム画面上のフィードバックカードとして表示する
- `バックアップ` タイル内のメニューから、`バックアップを書き出し` と `バックアップを読み込み` を選べる

### 6.7 チャンネル一覧画面

- タイトルは `チャンネル一覧`
- サブタイトルは選択中の並び順に追従する
- 先頭に `Tips` タイルを 1 枚置き、登録件数、並び順、`タップで動画一覧`、`長押しで削除` をまとめて表示する
- 単独画面レイアウトでは、チャンネルタイルを縦スクロールで並べる
- 分割レイアウトでは、左ペインに選択状態付きのチャンネルタイルを並べる
- 分割レイアウトの内容表示側には、選択中チャンネル名とその動画一覧を表示する

### 6.8 チャンネル別動画一覧画面

- タイトルはチャンネル名
- サブタイトルは `このチャンネルの動画を新しい順に最大50件表示`
- 各動画タイルにサムネイル、タイトル、チャンネル名、投稿日を表示
- 各動画タイルの右上には、チャンネル一覧画面と同じ番号バッジで 1 始まりの通し番号を表示する
- 各動画タイルの右下には `丸めた分数 + 再生数` を表示する

### 6.9 動画一覧画面

- タイトルは `動画一覧`
- サブタイトルは `キャッシュ済み動画を新しい順に最大50件表示`
- 各動画タイルにサムネイル、タイトル、チャンネル名、投稿日を表示
- 各動画タイルの右上には、チャンネル一覧画面と同じ番号バッジで 1 始まりの通し番号を表示する
- 各動画タイルの右下には `丸めた分数 + 再生数` を表示する

### 6.10 戻る操作

- 一覧画面は左端からの右スワイプで戻れること
- 戻る判定は `BackSwipePolicy` に従う
- 縦スクロールを極力阻害しないこと

理由:
- このアプリでは Safari のように、スクロールしたい時に指の接地条件に左右されず入力を拾えることを優先する。
- そのため独自の戻るジェスチャーは、左端の狭い範囲と十分大きい水平移動に限定し、通常の縦スクロールを優先する。

## 7. 非機能要件

### 7.1 起動性能

- 起動直後は軽量キャッシュのみを読み、ホーム画面到達を優先する
- 起動直後にネットワーク更新を自動実行してはならない
- 起動時の診断タイムラインを取得できること

### 7.2 操作性

- 一覧スクロール中に更新反映でカクつきや固まりを起こしにくいこと
- 動画を誤タップで開きにくいこと
- 戻るスワイプと縦スクロールの衝突を最小化すること
- スクロール入力は、指先でも親指の角質部分でも詰まりにくく、入力を拾う側に寄せること

### 7.3 永続性

- キャッシュはアプリ終了後も残ること
- サムネイルも再利用できること

### 7.4 拡張性

- 将来の検索、並び替え、フィルタ追加を前提にデータ構造を設計する
- UI とロジックを分離する
- 画面共通挙動は共通コンテナへ集約する

### 7.5 テスト性

- ネットワークを使わない UI テストができること
- モック起動時に fixture を seed できること
- 起動や更新の観測点を accessibility identifier または diagnostics timeline で読めること
