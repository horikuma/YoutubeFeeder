# HelloWorld Rules

この文書は、HelloWorld プロジェクトの上位方針を定める正本です。ここでは、実装や現行機能の詳細ではなく、長期に維持したい判断基準と変更管理ルールを扱います。機能仕様は [spec.md](spec.md)、実装構造と責務分担は [architecture.md](architecture.md) を参照してください。

## 文書の境界

- `rules.md` には、根幹普遍の方針、変更判断、文書運用、変更管理を記述する。
- [spec.md](spec.md) には、ユーザー向け機能、画面遷移、操作、表示要件を記述する。
- [architecture.md](architecture.md) には、実装構造、責務分担、データフロー、テスト配置、現在の採用方式を記述する。
- `rules.md` へ実装詳細や画面単位の仕様を集約してはならない。
- 文書の内容が詳細化しすぎた場合は、上位方針を残して詳細を `architecture.md` または `spec.md` へ移す。

## 目的

- 人と LLM のどちらが変更しても、同じ判断基準で継続開発できる状態を保つ。
- UI、ロジック、永続化、外部通信の責務境界を保ち、変更の影響範囲を局所化する。
- 起動性能、操作中の軽さ、保守性、テスト容易性を長期的に損なわない。

## 設計原則

- 画面は表示責務に集中し、ルール判断や I/O を直接持たない。
- ルールは UI から切り離し、再利用可能な pure logic として扱う。
- 永続化や外部通信は、UI から直接触らず、中間層を介して扱う。
- 端末差分やレイアウト差分は、機能差分ではなく表現差分として扱う。
- テストしやすさを優先し、観測可能な state や test marker を用意する。
- Apple が推奨するデザインおよび実装パターンは、特別な理由がない限り積極的に採用する。
- 独自実装や標準から外れた設計を選ぶ場合は、必要性と理由を説明できる状態にする。

## 変更時の判断ルール

- 新しい変更では、まず `表示責務`、`ルール`、`永続化または外部通信` を分けて考える。
- 2 画面以上で共有しうる処理は、早い段階で共通化を検討する。
- テストのためだけの分岐は、本番実装へ広げずテスト支援レイヤーへ閉じ込める。
- ネットワークやファイル I/O を View に直接書かない。
- actor 境界や main actor 隔離に関わる設計では、UI モデルと永続化モデルを混同しない。
- 既存文書の章構成を尊重し、関連するルールは適切な節へ整理して記述する。
- Git のコミットメッセージは日本語で記述する。

## テスト原則

- テストは `unit test` と `UI test` の責務を分ける。
- 純粋なルール、並び順、parser、resource 読込は `unit test` で担保する。
- 画面遷移、主要導線、操作体験、モック経路は `UI test` で担保する。
- UI テストでは実ネットワークを使わない。
- 回帰確認の標準ターゲットは 1 つに固定し、継続運用可能な形を優先する。
- ソースコードを変更した場合は、変更内容に応じたテストを実行し、通過を確認してからコミットする。
- ドキュメントのみを変更した場合は、試行錯誤の途中段階を含みやすいため、自動ではコミットせず指示を待つ。

## 文書運用ルール

- 機能を変更したら [spec.md](spec.md) を見直す。
- 実装構造、責務分担、採用アーキテクチャ、テスト配置を変更したら [architecture.md](architecture.md) を見直す。
- 上位方針や変更判断の基準を変更したら `rules.md` を見直す。
- 意識的な設計変更や、標準案と代替案の間で判断した内容は `decisions.md` に記録する。
- 実装と文書にずれが出た場合は、どちらが正しいか確認してから揃える。
- 一時的な事情や現状の実装詳細を、恒久ルールとして `rules.md` に固定しない。
- 通常のプロジェクト文書ファイル名は `lowercase-kebab-case.md` または `lowercase.md` に統一する。
- `README.md` や `CONTRIBUTING.md` のような広く定着した慣例名だけは例外として大文字を許容する。

## chatlog.md 運用ルール

- `chatlog.md` を更新せよ、という指示を受けた場合に更新する。
- 日付が変わっている場合は、新しい日付見出しをファイル先頭へ追加する。
- 同じ日付の中では、新しい発言ほど上、古い発言ほど下になるように追記する。
- 基本的にはユーザー発言をそのまま記録する。
- ただし、明らかな変換ミスや誤字は、意味を変えない範囲で修正してよい。
- `chatlog.md` は、LLM や Codex をどのように使って構築したかを後から振り返るための履歴として扱う。

## decisions.md 運用ルール

- 意識的な設計変更が行われた際に更新する。
- 日付見出しの下へ、新しい決定ほど上に追加する。
- 各決定は箇条書きで記述し、その直下の 1 段下げた行に理由を書く。
- `decisions.md` は、なぜその設計判断を選んだかを後から振り返るための履歴として扱う。

## 変更管理ルール

アプリ機能を変更する場合は、次を同じ変更セットで更新しなければならない。

- 機能仕様に関わる文書
- 影響を受ける実装
- 必要なテスト

また、次の変更は対応する文書更新を伴わなければならない。

- 機能変更: [spec.md](spec.md)
- 実装構造や責務変更: [architecture.md](architecture.md)
- 上位方針や運用変更: `rules.md`
