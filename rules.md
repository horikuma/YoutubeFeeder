# HelloWorld Rules

この文書は、HelloWorld プロジェクトの上位方針を定める正本です。ここでは、実装や現行機能の詳細ではなく、長期に維持したい判断基準と変更管理ルールを扱います。機能仕様は [spec.md](spec.md)、実装構造と責務分担は [architecture.md](architecture.md) を参照してください。

## 文書の境界

- `rules.md` には、根幹普遍の方針、変更判断、文書運用、変更管理を記述する。
- [spec.md](spec.md) には、ユーザー向け機能、画面遷移、操作、表示要件を記述する。
- [architecture.md](architecture.md) には、実装構造、責務分担、データフロー、テスト配置、現在の採用方式を記述する。
- `rules.md` へ実装詳細や画面単位の仕様を集約してはならない。
- 文書の内容が詳細化しすぎた場合は、上位方針を残して詳細を `architecture.md` または `spec.md` へ移す。

## 目的

- 人と LLM のどちらが変更しても、同じ判断基準で継続開発できる状態を保つ。
- UI、ロジック、永続化、外部通信の責務境界を保ち、変更の影響範囲を局所化する。
- 起動性能、操作中の軽さ、保守性、テスト容易性を長期的に損なわない。

## 設計原則

- 本アプリの基本モデルは `MVVM + Clean Architecture` とし、`View`、`ViewModel または coordinator`、`UseCase 相当の service`、`Store / Infrastructure` の依存方向を保つ。
- 画面は表示責務に集中し、ルール判断や I/O を直接持たない。
- ルールは UI から切り離し、再利用可能な pure logic として扱う。
- 永続化や外部通信は、UI から直接触らず、中間層を介して扱う。
- 端末差分やレイアウト差分は、機能差分ではなく表現差分として扱う。
- 画面をいたずらに増やさず、統一された操作感とフィードバック感を優先する。
- テストしやすさを優先し、観測可能な state や test marker を用意する。
- Apple が推奨するデザインおよび実装パターンは、特別な理由がない限り積極的に採用する。
- 独自実装や標準から外れた設計を選ぶ場合は、必要性と理由を説明できる状態にする。
- API キーやトークンなどの秘密情報は、リポジトリへコミットせず、ローカル専用かつ ignore 済みの設定ファイルから注入する。
- 外部 API の認証情報は、可能な限り URL クエリへ載せず、ログや履歴へ残りにくい経路を優先する。
- 外部 API 用のキーは用途ごとに分け、利用 API やアプリ種別で制限できる前提で扱う。

## 変更時の判断ルール

- 新しい変更では、まず `表示責務`、`ルール`、`永続化または外部通信` を分けて考える。
- 機能追加では、実装前に期待する振る舞いを表すテストを先に追加し、そのテストで失敗を確認してから実装へ進む。
- 不具合対応では、想定する原因候補ごとに再現または非再現を担保するテストを先に追加し、そのテストで失敗を確認してから修正へ進む。
- 2 画面以上で共有しうる処理は、早い段階で共通化を検討する。
- テストのためだけの分岐は、本番実装へ広げずテスト支援レイヤーへ閉じ込める。
- ネットワークやファイル I/O を View に直接書かない。
- actor 境界や main actor 隔離に関わる設計では、UI モデルと永続化モデルを混同しない。
- 既存文書の章構成を尊重し、関連するルールは適切な節へ整理して記述する。
- Git のコミットメッセージは日本語で記述する。
- 秘密情報がチャットへ貼られた場合は、その値を以後の記録文書へ転記せず、必要ならマスクして扱う。

## 実装健康度バロメタ

- 機能追加や障害修正では、動作確認だけでなく `責務境界`、`複雑度`、`リソースの作用域`、`MVVM の健全性` を同時に点検する。
- 点検の起点として `./scripts/health_barometer.sh` を使い、少なくとも `Swift ファイル行数`、`関数行数`、`1ファイル内の型数`、`ObservableObject の公開状態量` を継続観測する。
- `Swift ファイル行数` は原則 `500` 行未満を維持し、`900` 行以上は分割なしに進めてはならない。
- `関数行数` は原則 `60` 行未満を維持し、`80` 行以上は分割または責務移譲なしに進めてはならない。
- `1ファイル内の型数` が `12` を超える場合は、モデル、永続化、外部通信、UI 補助などが混在していないか見直す。
- `ObservableObject` や coordinator の `@Published` 状態は `8` 個前後を目安とし、増え続ける場合は別 service、store、または画面ローカル state へ戻すことを検討する。
- View は一時的な UI 状態、アニメーション状態、選択状態、ダイアログ状態を持ってよく、MVVM を理由にそれらを機械的に ViewModel へ押し上げてはならない。
- ViewModel または coordinator は `1 画面` もしくは `1 機能の orchestration` に責務を寄せ、画面描画専用の細かな値変換や単純な表示状態まで過剰に抱え込まない。
- `Clean Architecture` の境界を守るため、`ViewModel` や coordinator の内部で `Store` や `Service` を直接 new せず、composition root から依存注入する。
- 依存方向は `View -> ViewModel/coordinator -> service/use case -> Store or Infrastructure` を原則とし、内側の層が外側の UI 都合へ依存してはならない。
- MVVM を守るためだけの protocol 分割、値の受け渡しラッパー、薄い中継層は増やさず、代わりに `pure logic`、`store`、`external service` へ意味のある単位で分割する。
- 固定パス、永続ファイル、検索キャッシュ、秘密情報解決のような `スコープの広いリソース` は、専用の `Paths` / `Store` / `Service` 型へ閉じ込め、View や汎用 model ファイルへ散らしてはならない。
- `health_barometer` が警告を出した箇所は、今回変更で悪化していないかを確認し、悪化する場合は同じ変更セット内で整理する。

## テスト原則

- テストは `unit test` と `UI test` の責務を分ける。
- テストは後追いの確認手段ではなく、仕様を明確化し、仕様競合や成立不能条件を早期検出するための正本の一部として扱う。
- 純粋なルール、並び順、parser、resource 読込は `unit test` で担保する。
- 画面遷移、主要導線、操作体験、モック経路は `UI test` で担保する。
- UI テストでは実ネットワークを使わない。
- 回帰確認の標準ターゲットは 1 つに固定し、継続運用可能な形を優先する。
- UI テストは、重複する起動導線や確認項目を統合し、fixture や test support によって待ち時間を抑える。
- スワイプや pull-to-refresh の UI テストでは、実ジェスチャーで不安定になりやすい箇所に限り、test support の `ダミーシンボル` や専用 trigger によるイベント発火を許容する。
- ビルド確認では `error` だけでなく `warning` も 0 件であることを確認し、警告を残したまま完了扱いにしてはならない。
- 機能追加、不具合修正、機能削除のいずれでも、先に追加したテストや既存テストを変更後の仕様へ合わせて更新し、古い仕様を表すテストを放置してはならない。
- ソースコードを変更した場合は、変更内容に応じたテストを実行し、通過を確認してからコミットする。
- ドキュメントのみを変更した場合は、試行錯誤の途中段階を含みやすいため、自動ではコミットせず指示を待つ。
- ソースコードを含むコミットでは、検証前後に取得できる build 時間、test 時間、再試行回数、起動性能の観測値を `metrics.md` へ記録する。
- ドキュメントのみのコミットでも `metrics.md` は更新対象とし、実測を省略した理由を記録する。
- 起動性能は、少なくとも `起動からスプラッシュ表示まで`、`スプラッシュ表示からホーム表示まで`、`起動からホーム表示まで` を継続観測できる状態を保つ。
- 計測を手作業へ寄せすぎず、`xcodebuild` や UI テストから取得可能な値はコマンドで再取得できる形を優先する。
- ソースコード変更をコミットした直後は、ユーザーのヒューマンテストへつなげられるよう、アプリを起動済みの状態に整える。
- ヒューマンテストの既定デバイスは物理 `iPhone 12 mini` とする。
- `iPad` でのヒューマンテストへ切り替えるのは、ユーザーから明示的な指示があった場合に限る。

## 文書運用ルール

- `rules.md` は追記メモ置き場として扱わず、章ごとの役割が読んで辿れる構造を維持する。
- `rules.md` を更新する時は、関連する既存節へ統合し、重複、矛盾、末尾だけの場当たり的な追記を避ける。
- 機能を変更したら [spec.md](spec.md) を見直す。
- 実装構造、責務分担、採用アーキテクチャ、テスト配置を変更したら [architecture.md](architecture.md) を見直す。
- 上位方針や変更判断の基準を変更したら `rules.md` を見直す。
- 意識的な設計変更や、標準案と代替案の間で判断した内容は `decisions.md` に記録する。
- 検証コストや起動性能の観測値を取得したら `metrics.md` に記録する。
- 実装と文書にずれが出た場合は、どちらが正しいか確認してから揃える。
- 一時的な事情や現状の実装詳細を、恒久ルールとして `rules.md` に固定しない。
- 通常のプロジェクト文書ファイル名は `lowercase-kebab-case.md` または `lowercase.md` に統一する。
- `README.md` や `CONTRIBUTING.md` のような広く定着した慣例名だけは例外として大文字を許容する。

## chatlog.md 運用ルール

- `chatlog.md` は継続更新対象とし、ユーザー発言が追加されたら都度反映する。
- 日付が変わっている場合は、新しい日付見出しをファイル先頭へ追加する。
- 新しい日付見出しを追加する場合は、直前の日付見出しとの間に 1 行だけ空行を入れる。
- 同じ日付の中では、新しい発言ほど上、古い発言ほど下になるように追記する。
- 各発言には、直前の LLM 応答で表示された所要時間を `（直前のLLM所要時間: 約11m52s）` のような形式で併記する。
- 所要時間は厳密一致を要求せず、記録時点との差による数十秒のずれは許容する。
- 基本的にはユーザー発言をそのまま記録する。
- ただし、明らかな変換ミスや誤字は、意味を変えない範囲で修正してよい。
- `chatlog.md` が `Chatlog.md` や `Chatlog` のように揺れていても、明らかに既存の `chatlog.md` を指している場合はファイル名を `chatlog.md` にそろえて記録する。
- 個人情報が混ざっている場合は、`chatlog.md` へ書く前に除去し、その対応をチャットウィンドウ上でも明示する。
- API キーやトークンなどの秘密情報は、個人情報に準ずる情報として扱い、そのまま記録しない。
- 絶対パスやホームディレクトリを含む文字列は、個人情報に準ずる情報として扱い、`<workspace>/...` や相対パスへマスクして記録する。
- `chatlog.md` は、LLM や Codex をどのように使って構築したかを後から振り返るための履歴として扱う。

## decisions.md 運用ルール

- 意識的な設計変更が行われた際に更新する。
- 日付見出しの下へ、新しい決定ほど上に追加する。
- 新しい日付見出しを追加する場合は、直前の日付見出しとの間に 1 行だけ空行を入れる。
- 各決定は箇条書きで記述し、その直下の 1 段下げた行に理由を書く。
- `decisions.md` は、なぜその設計判断を選んだかを後から振り返るための履歴として扱う。

## 変更管理ルール

LLM は、ユーザーから実装指示を受けた場合、次の一連シーケンスを完了条件として扱う。

1. 指示内容を読み取り、関連コードと関連文書を確認する。
2. 想定原因や変更方針を分け、不具合対応では原因候補ごとの失敗テストを先に追加する。
3. 実装を変更し、責務境界と実装健康度バロメタの悪化がないか点検する。
4. 変更内容に応じたテストと build 検証を行い、`error 0` かつ `warning 0` を確認する。
5. 必要な文書 `spec.md`、`architecture.md`、`decisions.md`、`metrics.md`、`chatlog.md`、`rules.md` を更新する。
6. ソースコード変更を含む場合は、ユーザーから明示的に止められていない限り、最後にコミットまで実施する。

コミット時の原則は次の通り。

- コミット対象は、その時点で完了条件を満たした変更セット全体とする。
- ソースコード変更を含む場合は、テスト、文書更新、計測記録を含めてからコミットする。
- コミットメッセージは日本語で記述する。
- ユーザーが一括コミットを許可した場合は、個別分割よりも完了状態の一括コミットを優先してよい。

アプリ機能を変更する場合は、次を同じ変更セットで更新しなければならない。

- 機能仕様に関わる文書
- 影響を受ける実装
- 必要なテスト
- 必要な計測記録

また、次の変更は対応する文書更新を伴わなければならない。

- 機能変更: [spec.md](spec.md)
- 実装構造や責務変更: [architecture.md](architecture.md)
- 上位方針や運用変更: `rules.md`
- 検証コストや性能観測の更新: [metrics.md](metrics.md)
