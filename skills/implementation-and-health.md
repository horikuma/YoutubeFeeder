# Implementation And Health Skill

この文書は、実装と健康度点検という 1 つのタスクだけを定義する。

## 実装と健康度点検

- 実装と健康度点検とは、変更を実装しながら、責務境界、複雑度、リソースの作用域、状態管理の健全性を同時に点検し、悪化を残さない状態へ整えるタスクである。
- このタスクでは、実装、健全性観測、試行回数の自己診断、停止判断までをこの文書だけで判断しなければならない。

## 実施内容

- 実装中は、責務境界、複雑度、リソースの作用域、状態管理の健全性を同時に点検し、悪化する変更を放置してはならない。
- テストのためだけの分岐は、本番実装へ広げずテスト支援レイヤーへ閉じ込めなければならない。
- UI 導線から起動される機能でも、domain または logic 側へ切り出せる判定や状態遷移がある場合は、まずその層へ寄せることを検討しなければならない。
- 健全性の観測では、少なくとも `ファイル行数`、`関数行数`、`1ファイル内の型数`、`公開状態量` を確認しなければならない。
- `Swift ファイル行数` は原則 `500` 行未満を維持し、`900` 行以上は分割なしに進めてはならない。
- `関数行数` は原則 `60` 行未満を維持し、`80` 行以上は分割または責務移譲なしに進めてはならない。
- `1ファイル内の型数` が `12` を超える場合は、モデル、永続化、外部通信、UI 補助などが混在していないか見直さなければならない。
- `ObservableObject` または coordinator の `@Published` 状態は `8` 個前後を目安とし、増え続ける場合は別 service、store、または画面ローカル state への分離を検討しなければならない。
- 契約ログを追加する場合は、起点、終点、正常パス、異常パスを同じ観測系列で追跡できるようにし、どの条件で開始し、どこで終了し、どこで正常または異常に分岐したかを後から一意に辿れる状態で実装しなければならない。
- 契約ログの観測系列が 1 つでも欠けている場合は、原因切り分けに使えない前提で扱い、観測点の追加か責務移譲を先に検討しなければならない。
- `LLM試行回数` は、1つのユーザー指示に対して `仮説 -> 実装またはテスト -> 失敗 -> 修正` を何巡したかとして数えなければならない。
- `同一論点の再試行回数` は、同じ失敗原因に対してテストや実装の修正をやり直した回数として数えなければならない。
- TDD のために先に追加した失敗するべき新規テストを、その後の実装で通した `赤 -> 緑` の 1 往復は、`LLM試行回数` と `同一論点の再試行回数` に含めてはならない。
- `LLM試行回数 >= 4` または `同一論点の再試行回数 >= 3` に達した場合は、新規実装を中止し、観測内容と停止理由をユーザーへ報告しなければならない。
- 秘密情報がチャットへ貼られた場合は、その値を以後の記録文書へ転記してはならず、必要ならマスクして扱わなければならない。
- `llm-temp/` 配下の一時ファイルは、ユーザーが後から処理経路を見直すための痕跡として扱い、LLM の判断で自動削除してはならない。

## 完了条件

- 実装結果に加えて、責務境界、複雑度、リソースの作用域、状態管理の健全性を確認していること。
- 契約ログを実装した場合は、起点、終点、正常パス、異常パスが 1 つの系列で観測できることを確認していること。
- 観測対象の数値と状態を確認し、しきい値超過がある場合は分割、責務移譲、停止のいずれかを実施していること。
- 試行回数の自己診断を行い、しきい値超過時は新規実装を停止して報告していること。
- 秘密情報と一時ファイルの扱いがこの文書の規則どおりであること。

## 禁止事項

- 健全性悪化を認識したまま実装を続行してはならない。
- しきい値超過を無視して新規実装を進めてはならない。
- テスト専用分岐を本番実装へ残してはならない。
- 秘密情報を記録文書へ転記してはならない。
- `llm-temp/` 配下の一時ファイルを LLM の判断だけで削除してはならない。
