# User Instruction Understanding Skill

この文書は、ユーザー指示の理解という 1 つのタスクだけを定義する。

## ユーザー指示の理解

- ユーザー指示の理解とは、1 件のユーザー指示について、変更対象、影響範囲、関連コード、関連文書、未確定事項を確定し、以後の実装や文書更新で参照する作業境界を整えるタスクである。
- このタスクでは、ユーザー指示の読取り、変更種別の判定、必要なコード読込み、必要な文書読込み、影響範囲の確定、曖昧さの検出までをこの文書だけで判断しなければならない。
- この文書で使う usage 記法では、角括弧 `[...]` 内は省略可能部分を表す。
- この文書で使う usage 記法では、角括弧の外にある要素は必須であり、左から右の順にそのまま指定しなければならない。
- この文書で使う usage 記法では、山括弧 `<...>` 内は実行時に具体値へ置換して渡す値を表す。

## 実施内容

- チャット入力でユーザー指示を受けた場合は、以後のタスクを始める前に、このタスクを最初のアクションとして実施しなければならない。
- `skills/hooks/on-input.md` で確定した `user_line` は、このタスクで扱う 1 行のユーザー指示の正本として扱わなければならない。
- 着手時は、対象となるユーザー指示を読み、その指示が要求している変更結果を 1 件に確定しなければならない。
- 着手時は、このリポジトリで用意済みのタスク群のうち今回の指示に関係するものを列挙し、各タスクを実施する前に対応するルール文書を読み込まなければならない。
- ユーザー指示に複数の要求が含まれる場合は、その全要求が同じ変更対象へ属するかを確認しなければならない。
- ユーザー指示から、少なくとも変更対象、期待される結果、明示されている制約、明示されていない部分を確定しなければならない。
- 変更対象の確認には、対象コード、対象文書、対象設定のうち、ユーザー指示の達成に必要なものだけを読まなければならない。
- 影響範囲の確認では、変更対象から直接参照される実装、依存先、出力先、更新先のうち、今回の変更結果に影響するものだけを読まなければならない。
- 関連しないコード、関連しない文書、参考情報、将来のための先回り読込みをしてはならない。
- ユーザー指示が機能追加、画面仕様変更、画面遷移変更、表示要件変更のいずれかに該当する場合は、`docs/specs/specs-product.md` を読まなければならない。
- ユーザー指示がレイヤ構造、依存方向、データフロー、責務境界の変更に該当する場合は、`docs/specs/specs-architecture.md` を読まなければならない。
- ユーザー指示がファイル単位、型単位、テスト単位の責務や構成の変更に該当する場合は、`docs/specs/specs-design.md` を読まなければならない。
- ユーザー指示がローカル開発環境、版固定、セットアップ、再現性の変更に該当する場合は、`docs/specs/specs-environment.md` を読まなければならない。
- ユーザー指示が文書配置や文書更新先の判断を含む場合は、`docs/rules.md` の `文書同期` 節を読まなければならない。
- ユーザー指示が `skills`、`scripts`、`tools` の変更に該当する場合は、`docs/rules.md` の `スキル作成` 節を読まなければならない。
- ユーザー指示がルール文書の作成または更新に該当する場合は、`docs/rules.md` の `ルール作成・更新` 節を読まなければならない。
- ユーザー指示が、skill の呼び出し方、`scripts` 入口、`llm-cache` 参照キーの rules への固定に該当する場合は、`docs/rules.md` に加えて、対応する `scripts/*`、`scripts/*/_meta.json`、`llm-cache/*.json` を読み、command 形式と参照キー名を確定しなければならない。
- Issue に基づいて着手する場合は、実装前に対象 Issue の Description を読み、作業単位、完了条件、非対象を確定しなければならない。
- 上記のどの種別にも該当しない場合は、追加の文書を読まず、ユーザー指示と直接関係するコードや既存ファイルだけを確認しなければならない。
- タスク規定が曖昧で、変更対象、完了条件、非対象、更新先のいずれかを確定できない場合は、処理を中断し、その旨をユーザーへ報告しなければならない。
- `llm-cache` を参照する rules を更新する場合は、値を転記してはならず、`session-context.json` の `sessionMainBranch`、`github-app.json` の `operationMode`、`issue-defaults.json` の `assignee.login` のように、参照するキー名だけを記述しなければならない。

## 完了条件

- ユーザー指示について、変更対象と影響範囲が確定していること。
- 今回の指示に関係する用意済みタスクと、その実施前に読むルール文書が確定していること。
- ユーザー指示の達成に必要なコードと文書だけを読んでいること。
- 追加で読むべき文書がある場合は、この文書の条件に基づいて選択されていること。
- command 形式や `llm-cache` 参照キーを rules へ固定する指示では、その判断に必要な `scripts`、`_meta.json`、`llm-cache` だけを読んでいること。
- 曖昧さが残る場合は、実装へ進まず中断報告していること。

## 禁止事項

- ユーザー指示と無関係なコードや文書を参考目的で読んではならない。
- 用意済みタスクの列挙や実施前ルール読込みを省略したまま、後続タスクへ進んではならない。
- 変更種別に該当しない `specs` や `rules` を慣習で読んではならない。
- 変更対象や影響範囲を推測や慣習で補完してはならない。
- 完了条件や非対象を確定しないまま後続タスクへ進んではならない。
- rules に、usage、必要な補足、具体的な command 例文だけで実行方法を確定できない command 例や、値を露出した `llm-cache` 記述を残してはならない。
